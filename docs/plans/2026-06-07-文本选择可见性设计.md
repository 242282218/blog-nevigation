# 文本选中态可见性优化设计

日期：2026-06-07

## 背景

当前全站选中文字使用 `src/app/globals.css` 中的全局 `::selection`：背景为 `--terracotta-300`，文字为 `--surface-elevated`。在浅色界面和部分内容区域里，选区对比度不够明显，用户选中文本或内容时不容易判断选区范围。

## 目标

- 统一提升全站文本选中态的可见性。
- 保持现有暖色品牌风格，不引入新的视觉体系。
- 只改全局选区样式和对应设计 token，不改组件结构。

## 方案比较

### 方案 A：直接加深全局 `::selection` 颜色（推荐）

在设计 token 中增加语义化 selection token，并让全局 `::selection` 使用深色 terracotta 背景与白色文字。

优点：改动最小，覆盖全站，行为一致，后续维护简单。
缺点：只能解决浏览器文本选区，不能改变自定义组件的非文本选中态。

### 方案 B：按区域分别覆盖选区颜色

为文章、编辑器、导航卡片分别写更细的 `::selection` 覆盖。

优点：每个区域可微调。
缺点：规则分散，容易互相覆盖，后续新增区域可能遗漏。

### 方案 C：同时增加焦点/选中视觉系统

在文本选区之外，统一梳理焦点、卡片选中、表单选中等所有状态。

优点：可形成完整交互状态规范。
缺点：范围扩大，容易触碰未请求的 UI 行为，不符合本次小修目标。

## 选定设计

采用方案 A。

新增语义 token：

- `--selection-bg`: 选区背景，使用 `--terracotta-700`。
- `--selection-fg`: 选区文字，使用 `#ffffff`。

全局 `::selection` 只引用这两个 token。这样把“选区语义”放入设计 token 层，而不是在 `globals.css` 里硬编码品牌色。

## 架构与组件影响

- `src/app/styles/design-tokens.css`：新增 Selection 语义 token。
- `src/app/globals.css`：更新全局 `::selection` 使用语义 token。
- 不修改 React 组件、不新增交互逻辑、不改变编辑器选择行为。

## 数据流与错误处理

本改动是 CSS 表现层改动，没有数据流和运行时错误处理变化。浏览器不支持自定义属性时会沿用 CSS 解析规则；项目已依赖 CSS 变量作为设计 token，因此无需额外 fallback。

## 验证

- 运行 lint，确认 CSS/TS 配置不受影响。
- 如有条件，可启动页面手动选中文章、导航卡片、编辑器文本，确认选区背景明显、文字可读。
