# 剪切板图片粘贴与 R2 媒体备份测试方案

## 文档信息

- 编写日期：2026-06-17
- 适用功能：文章编辑器剪切板图片粘贴、本地媒体目录、R2 媒体对象同步、R2 备份恢复
- 关联设计文档：`docs/plans/2026-06-17-剪切板图片粘贴与R2媒体备份设计.md`
- 关联运行时契约：`docs/API接口契约与运行时数据格式.md`

## 测试目标

本测试方案要证明四件事：

1. 文章编辑时可以直接从剪切板粘贴图片，并稳定插入 Markdown 图片链接。
2. 图片文件先落到本地持久化媒体目录，Docker 更新和服务重启不丢失。
3. 图片通过现有 R2 自动配置同步为独立媒体对象，不引入第二套 R2 媒体配置。
4. R2 JSON 备份继续保持明文 JSON，能直接 `JSON.parse`，不包含图片二进制、base64、加密字段或加密口令。

## 测试原则

- 先测根因链路：粘贴、存储、清单、R2 对象、恢复，而不是只看 UI 是否显示成功。
- 每个写入动作必须验证磁盘结果、响应体和备份副作用。
- R2 相关测试必须同时验证正向行为和项目级禁忌行为。
- 自动化优先覆盖稳定逻辑；真实 R2、Docker 更新和浏览器剪切板作为集成/验收补充。
- 所有测试必须使用临时数据目录，不能污染生产 `data/`。
- 涉及 R2 的测试默认使用 mock；真实 R2 只在受控测试 bucket 中跑。
- 测试失败要能定位到层级：前端、API、存储、R2、恢复、部署。

## 必守约束

### R2 备份约束

以下断言属于发布阻断项：

- 不新增 `R2_MEDIA_BUCKET`、`R2_MEDIA_PREFIX`、`R2_MEDIA_ACCESS_KEY_ID`、`R2_MEDIA_SECRET_ACCESS_KEY` 等媒体专用配置。
- 不新增 `R2_BACKUP_ENCRYPTION_PASSPHRASE` 或任何 R2 自动备份加密口令字段。
- `latest/backup.json` 和 `snapshots/*.json` 必须是明文 JSON。
- R2 JSON 备份体必须可直接 `JSON.parse`。
- JSON 备份中不能出现 `data:image`、常见 base64 图片头、图片原始字节转码内容。
- 图片必须作为独立对象写入 `<R2_PREFIX>/media/files/yyyy/mm/<sha256>.<ext>`。

### Docker 数据约束

以下断言属于发布阻断项：

- Docker 更新脚本必须保留宿主机 `.env`。
- Docker 更新脚本必须保留宿主机 `data/`。
- 媒体目录必须位于宿主机持久化挂载路径下，例如 `/opt/blog-nevigation/data/media/files/`。
- 更新前本地快照必须包含 `.env` 和 `data/`，因此也包含 `data/media/`。

### 安全约束

以下断言属于发布阻断项：

- `POST /api/data/media` 必须要求编辑器 session。
- `POST /api/data/media` 必须要求同源和 CSRF header。
- 用户不能提交最终存储路径。
- 公开读取 `/media/files/...` 不能路径穿越到 `data/media/files/` 外。
- 响应不能泄露本地绝对路径、R2 Secret、session token 或 CSRF token。

## 测试范围

### 范围内

- `MarkdownEditor` 粘贴事件处理。

- `src/lib/editor-media-remote.ts` R2 媒体恢复。
- 本地恢复和 R2 恢复接口的媒体统计与 warning。
- README、API 契约、部署文档与项目规则的一致性。

### 范围外

第一版不测试以下未实现能力：

- 媒体库 UI。
- 图片裁剪、压缩、格式转换。
- 孤儿图片自动删除。
- R2 公网直链。
- 多 bucket 或媒体专用 R2 凭证。
- 复杂多人协同编辑冲突合并。

## 单元测试方案

### U1：媒体类型识别

目标：

- 后端用 magic bytes 判断真实图片类型，不只信任 MIME。

覆盖文件：

- `tests/lib/editor-media-storage.test.ts`

测试用例：

| 编号 | 输入 | 期望 |
| --- | --- | --- |
| U1-1 | PNG magic bytes | `mimeType=image/png`，扩展名 `.png` |
| U1-2 | JPEG magic bytes | `mimeType=image/jpeg`，扩展名 `.jpg` |
| U1-3 | WebP magic bytes | `mimeType=image/webp`，扩展名 `.webp` |
| U1-4 | GIF87a | `mimeType=image/gif`，扩展名 `.gif` |
| U1-5 | GIF89a | `mimeType=image/gif`，扩展名 `.gif` |
| U1-6 | 文本字节 `ABC` | 抛出 `EditorMediaInvalidFileError` |
| U1-7 | 空字节 | 抛出 `EditorMediaInvalidFileError` |

断言：

- 生成的 `asset.mimeType` 与 magic bytes 匹配。
- 生成的 `asset.path` 扩展名与类型匹配。
- 非图片不会写入 `data/media/`。

### U2：单文件大小限制

目标：

- 防止大图片拖垮 API、R2 和 JSON 备份协调流程。

测试用例：

| 编号 | 输入 | 期望 |
| --- | --- | --- |
| U2-1 | `5 * 1024 * 1024` 字节合法 PNG | 允许 |
| U2-2 | `5 * 1024 * 1024 + 1` 字节合法 PNG | 抛出 `EditorMediaFileTooLargeError` |

断言：

- 413 API 响应包含 `code=media_too_large`。
- 错误响应包含 `limitBytes`。
- 超大文件不写入本地文件。
- 超大文件不更新 manifest。
- 超大文件不触发 R2 上传。

### U3：hash 命名和去重

目标：

- 相同图片重复粘贴只保存一份文件。

测试步骤：

1. 在临时 `BLOG_DATA_ROOT` 上传 PNG。
2. 再次上传完全相同 PNG。
3. 读取 `data/media/manifest.json`。

断言：

- 两次返回的 `asset.id` 相同。
- 两次返回的 `asset.path` 相同。
- 第一次 `created=true`。
- 第二次 `created=false`。
- manifest 中只有一条 asset。
- 文件路径符合 `files/yyyy/mm/<sha256>.png`。
- `publicPath` 符合 `/media/files/yyyy/mm/<sha256>.png`。

### U4：manifest 解析和防御

目标：

- 防止损坏或恶意 manifest 导致路径逃逸或恢复异常。

测试用例：

| 编号 | manifest 内容 | 期望 |
| --- | --- | --- |
| U4-1 | 合法空清单 | 解析成功 |
| U4-2 | 合法单图片清单 | 解析成功 |
| U4-3 | `version` 错误 | 解析失败 |
| U4-4 | `assets` 不是数组 | 解析失败 |
| U4-5 | `path=../secret.png` | 解析失败 |
| U4-6 | `path=/absolute.png` | 解析失败 |
| U4-7 | `path=files/2026/../secret.png` | 解析失败 |
| U4-8 | `publicPath` 与 `path` 不一致 | 解析失败 |
| U4-9 | `hash` 不是 64 位 hex | 解析失败 |
| U4-10 | `mimeType=image/svg+xml` | 解析失败 |

断言：

- 非法 manifest 不能被恢复写入。
- `readEditorMediaManifest()` 遇到损坏文件抛出 `EditorMediaManifestInvalidError`。

### U5：媒体文件恢复校验

目标：

- 从 R2 下载媒体后必须校验 hash，防止写入错误文件。

测试用例：

| 编号 | 输入 | 期望 |
| --- | --- | --- |
| U5-1 | 下载字节 hash 与 manifest 一致 | 写入文件 |
| U5-2 | 下载字节 hash 不一致 | 抛出 `EditorMediaInvalidFileError` |
| U5-3 | 本地已有且 hash 一致 | 跳过下载 |
| U5-4 | 本地已有但 hash 不一致 | 下载并覆盖为正确文件 |

断言：

- hash mismatch 时不留下损坏文件。
- 恢复结果统计 `restored/skipped/failed` 正确。

### U6：R2 媒体 key 生成

目标：

- 媒体对象不能覆盖 JSON 备份对象。

覆盖文件：

- `tests/lib/r2-backup-storage.test.ts`

测试用例：

| 编号 | prefix | asset.path | 期望 key |
| --- | --- | --- | --- |
| U6-1 | `blog-navigation` | `files/2026/06/a.png` | `blog-navigation/media/files/2026/06/a.png` |
| U6-2 | `/custom-prefix/` | `files/2026/06/a.png` | `custom-prefix/media/files/2026/06/a.png` |
| U6-3 | `nested/prefix` | `files/2026/06/a.png` | `nested/prefix/media/files/2026/06/a.png` |

断言：

- `latest/backup.json` key 不变。
- `snapshots/...json` key 不变。
- 媒体对象 key 固定在 `<prefix>/media/` 下。

### U7：R2 JSON 明文不变

目标：

- 新增媒体清单后，JSON 备份仍可直接解析，不包含图片内容。

测试步骤：

1. 构造包含 `data.media.assets` 的 `EditorBackupPayload`。
2. 调用 `uploadBackupPayloadToR2()`。
3. 读取 mock S3 `PutObjectCommand.input.Body`。

断言：

- `ContentType=application/json; charset=utf-8`。
- `JSON.parse(body)` 成功。
- `JSON.parse(body).data.media.assets` 存在。
- `body` 不包含 `data:image`。
- `body` 不包含 `iVBOR`。
- `body` 不包含 `ciphertext`。
- `body` 不包含 `backupEncryption`。
- `body` 不包含任何加密口令字段。

### U8：R2 媒体上传对象

目标：

- 图片作为独立 R2 object 上传，ContentType 正确，Body 是 bytes。

测试步骤：

1. 配置 mock R2 env。
2. 构造 PNG `EditorMediaAsset`。
3. 调用 `uploadMediaAssetToR2(asset, bytes)`。

断言：

- 返回 `{ enabled: true, success: true, key }`。
- `PutObjectCommand.input.Key` 是 `<prefix>/media/<asset.path>`。
- `PutObjectCommand.input.ContentType` 是 `asset.mimeType`。
- `PutObjectCommand.input.Body` 是 `Buffer` 或二进制对象。
- 不修改 `latestUploadedBody`。

### U9：R2 未配置行为

目标：

- R2 关闭时不阻断本地媒体上传。

测试步骤：

1. 清空 R2 env 和配置文件。
2. 调用 `uploadMediaAssetToR2(asset, bytes)`。

断言：

- 返回 `{ enabled: false, success: false }`。
- 不创建 S3Client。
- 不发送 PutObjectCommand。
- API 层仍返回上传成功。

## API 测试方案

### A1：媒体上传鉴权

覆盖接口：

- `POST /api/data/media`

测试用例：

| 编号 | 请求 | 期望 |
| --- | --- | --- |
| A1-1 | 无 session | 401 |
| A1-2 | 有 session 无 CSRF | 403 |
| A1-3 | 有 session CSRF 不匹配 | 403 |
| A1-4 | 有 session Origin 不匹配 | 403 |
| A1-5 | 有 session CSRF Origin 都正确 | 进入上传逻辑 |

断言：

- 鉴权失败不读取 request body。
- 鉴权失败不写入本地文件。
- 鉴权失败不调用 R2。
- 鉴权失败不加入远端备份队列。

### A2：媒体上传成功

测试步骤：

1. 设置 `EDITOR_ACCESS_TOKEN`。
2. 设置临时 `BLOG_DATA_ROOT`。
3. 创建认证请求，带 CSRF。
4. 使用 `Content-Type: image/png` 原始 body 上传。
5. 读取响应 JSON。
6. 读取本地文件和 manifest。

断言：

- HTTP 200。
- `payload.success=true`。
- `payload.asset.path` 匹配 `files/yyyy/mm/<sha256>.png`。
- `payload.asset.publicPath` 匹配 `/media/files/...`。
- 本地文件存在。
- 本地文件字节与上传字节一致。
- manifest 中存在该 asset。
- `uploadMediaAssetToR2` 被调用。
- `queueCurrentBackupToRemote({ reason: 'media-write', writeSnapshot: false })` 被调用。

### A3：multipart 上传成功

测试步骤：

1. 构造 `FormData`。
2. `form.set('file', file)`。
3. 调用 `POST /api/data/media`。

断言：

- 与 A2 相同。
- 文件名不参与最终路径生成。
- 即使上传文件名包含 `../evil.png`，最终路径仍由 hash 生成。

### A4：请求体格式错误

测试用例：

| 编号 | 请求 | 期望 |
| --- | --- | --- |
| A4-1 | `Content-Type: application/json` | 400 |
| A4-2 | multipart 但无 `file` 字段 | 400 |
| A4-3 | multipart `file` 是字符串 | 400 |
| A4-4 | body 读取失败 | 400，提示请求格式无效 |

断言：

- 不写入文件。
- 不更新 manifest。
- 不调用 R2。

### A5：非法图片

测试用例：

| 编号 | body | 期望 |
| --- | --- | --- |

## 回归风险矩阵

| 风险 | 影响 | 触发场景 | 必测用例 |
| --- | --- | --- | --- |
| R2 JSON 备份被图片撑大 | R2 备份失败，恢复困难 | 把 base64 写入 backup payload | U7、B1、B6 |
| 媒体丢失 | 文章图片 404 | Docker 更新、R2 恢复 | D1、D2、R3 |
| R2 配置复杂化 | 用户配置成本上升 | 新增媒体专用 env | S4 |
| 路径穿越 | 数据泄漏 | 恶意 media path | U4、A10、SEC1 |
| MIME 伪造 | XSS 或非法文件托管 | 上传 SVG/HTML | U1、A5、SEC2 |
| 粘贴失败破坏正文 | 用户内容丢失 | 上传异常、异步替换错位 | C3、C4 |
| R2 恢复覆盖新数据 | 数据丢失 | 旧页面恢复 | R7 |
| R2 媒体对象缺失 | 图片无法恢复 | 手动删对象或同步失败 | R4、IR3 |
| 生产构建失败 | 无法发布 | Next route 类型错误 | S3 |

## 故障定位手册

### 上传返回 401

检查：

- 是否设置 `EDITOR_ACCESS_TOKEN`。
- 登录接口是否返回 session cookie。
- 请求是否带 `editor_session` cookie。

### 上传返回 403

检查：

- 请求是否带 `Origin`。
- `Origin` 是否等于运行时 public origin。
- CSRF cookie 和 header 是否一致。

### 上传返回 400 unsupported_media

检查：

- body 是否是真实图片字节。
- 测试中不要用不兼容的 Blob 实现导致字节被改写。
- magic bytes 是否完整。

### 上传返回 413

检查：

- 文件是否超过 5 MiB。
- 是否需要压缩后重试。

### 本地上传成功但 R2 失败

检查：

- `remoteMedia.enabled`。
- `remoteMedia.success`。
- R2 settings 文件是否损坏。
- R2 token 是否有 bucket item write 权限。
- prefix 是否正确。

### 图片访问 404

检查：

- Markdown 链接是否是 `/media/files/...`。
- `data/media/files/...` 是否存在。
- `data/media/manifest.json` 是否有该 asset。
- 文件 hash 是否和 manifest 一致。

### R2 恢复后图片缺失

检查：

- `mediaRestore.failed`。
- `mediaRestore.failures`。
- R2 上是否存在 `<prefix>/media/<asset.path>`。
- 下载字节 hash 是否匹配。

### latest JSON 解析失败

检查：

- R2 object ContentType 是否为 JSON。
- object body 是否被错误写成二进制。
- 是否错误把媒体文件上传到了 `latest/backup.json`。
