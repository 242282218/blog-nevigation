# 阶段零数据容灾设计

## 目标

按 `docs/快速开始.md` 阶段零修复数据丢失风险：首次启动必须配置 R2 或明确接受风险；写入后的远端备份任务不能只停留在内存；R2 对象必须加密；恢复前本地备份必须落盘；历史审计文档迁入归档并建立 ADR 目录。

## 方案

采用完整落地方案：R2 连接配置和加密口令继续保存到现有 `data/settings/cloudflare-r2.json`，让 Setup 页和设置页使用同一份热加载配置。旧配置文件兼容读取，缺少 `backupEncryptionPassphrase` 时视为未配置；启用 R2 时必须提供该口令，上传失败时返回明确错误。

## 组件边界

- `SetupWizard` 和 `/api/setup` 负责首次启动时的 R2 配置、加密口令和风险跳过确认。
- `r2-backup-storage` 负责 R2 配置读写、加密上传和兼容明文恢复。
- `backup-encryption` 负责 AES-256-GCM 加密格式、随机盐、解密和明文识别。
- `backup-coordinator` 负责 `${BLOG_DATA_ROOT}/.backup-pending.json` 的原子落盘、重启恢复、重试和 drain。
- `editor-remote-backup` 保留现有 `queueCurrentBackupToRemote` 签名，内部委托协调器。
- `startup-tasks` 在服务端启动路径触发 pending drain，避免等待下一次写入。
- `editor-data-storage` 在恢复前备份文件 copy 后执行 fsync。

## 数据流

写入 API 成功后继续调用 `queueCurrentBackupToRemote`。该函数检查 R2 状态后把任务写入 `.backup-pending.json`，再异步 drain。drain 每次读取当前编辑数据，调用 R2 上传。上传前把 JSON payload 加密为带 magic/version/header 的 JSON 对象；下载时先判断是否为加密对象，若不是则按旧明文 JSON 解析。

## 错误处理

R2 未启用时不入队；配置损坏返回原有错误。pending task 失败后增加 `retries`，最多 3 次，使用指数退避；达到上限后从 pending 文件移除并记录错误，避免永久卡住 drain。启用 R2 但缺少加密口令时上传失败并记录 ERROR。

## 测试

新增或更新单测覆盖：

- Setup 配置 R2 和跳过风险两条路径。
- pending 文件原子落盘、重启 drain、重试上限、并发入队。
- AES-256-GCM 加密/解密、随机盐、错误密钥失败、明文兼容。
- R2 上传对象不包含明文敏感内容，下载能恢复原 payload。
- 恢复前备份 copy 后调用 fsync。

最终以 `npm run check` 作为阶段零质量门禁。
