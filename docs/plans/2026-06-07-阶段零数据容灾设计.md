# 阶段零数据容灾设计

## 目标

按 `docs/快速开始.md` 阶段零修复数据丢失风险：首次启动必须配置 R2 或明确接受风险；写入后的远端备份任务不能只停留在内存；R2 对象必须固定为明文 JSON；恢复前本地备份必须落盘；历史审计文档迁入归档并建立 ADR 目录。

## 方案

采用完整落地方案：R2 连接配置继续保存到现有 `data/settings/cloudflare-r2.json`，让 Setup 页和设置页使用同一份热加载配置。R2 备份对象固定写入明文 JSON，配置只保存连接凭证和对象前缀。

## 组件边界

- `SetupWizard` 和 `/api/setup` 负责首次启动时的 R2 配置和风险跳过确认。
- `r2-backup-storage` 负责 R2 配置读写、明文 JSON 上传和明文 JSON 恢复。
- R2 自动备份链路不得要求备份口令；如需独立加密导出，只能放在独立工具链，不影响 R2 自动备份。
- `backup-coordinator` 负责 `${BLOG_DATA_ROOT}/.backup-pending.json` 的原子落盘、重启恢复、重试和 drain。
- `editor-remote-backup` 保留现有 `queueCurrentBackupToRemote` 签名，内部委托协调器。
- `startup-tasks` 在服务端启动路径触发 pending drain，避免等待下一次写入。
- `editor-data-storage` 在恢复前备份文件 copy 后执行 fsync。

## 数据流

写入 API 成功后继续调用 `queueCurrentBackupToRemote`。该函数检查 R2 状态后把任务写入 `.backup-pending.json`，再异步 drain。drain 每次读取当前编辑数据，调用 R2 上传。上传体必须是可直接 `JSON.parse` 的明文 JSON；下载时也直接按明文 JSON 解析，方便检查、迁移和恢复。

## 错误处理

R2 未启用时不入队；配置损坏返回原有错误。pending task 失败后增加 `attempts`、`lastAttemptAt` 和 `lastError`；达到上限后保留为 `failed` 状态，避免静默丢失，并允许用户在设置页手动重试。

## 测试

新增或更新单测覆盖：

- Setup 配置 R2 和跳过风险两条路径。
- pending 文件原子落盘、重启 drain、重试上限、并发入队。
- R2 上传体可直接 `JSON.parse`，`ContentType` 为 `application/json; charset=utf-8`，下载能恢复原 payload。
- R2 配置、API、UI 和文档不包含旧口令式远端备份字段。
- 恢复前备份 copy 后调用 fsync。

最终以 `npm run check` 作为阶段零质量门禁。
