# R2 自动配置设计

## 目标

在后台设置页提供“一键配置 R2”能力。管理员只输入 Cloudflare 登录邮箱、Global API Key、Account ID 和目标 bucket 名称，系统自动创建或复用 R2 bucket，创建该 bucket 专用的 R2 S3 凭证，并保存到现有 `BLOG_DATA_ROOT/settings/cloudflare-r2.json`。

## 方案

采用后台设置页请求后端执行自动配置。Global API Key 只存在于本次请求体和后端内存中，不写入设置文件、不回显到页面、不进入日志。自动配置完成后只保存应用运行所需的 R2 专用字段：

- `enabled`
- `accountId`
- `bucket`
- `accessKeyId`
- `secretAccessKey`
- `prefix`
- `endpoint`
- `snapshotOnWrite`
- 自动生成的备份加密密钥

现有手动配置继续保留，适合已有 R2 S3 凭证的场景。

## 数据流

1. 设置页提交 Cloudflare bootstrap 凭据和 bucket 名称。
2. 后端校验编辑会话、CSRF 和 `BLOG_DATA_ROOT`。
3. 后端调用 Cloudflare API 验证账号可访问。
4. 后端查询 R2 bucket 列表；存在则复用，不存在则创建。
5. 后端创建只绑定该 bucket 的 R2 对象读写 token。
6. 后端按 Cloudflare R2 S3 规则派生 `secretAccessKey`。
7. 后端生成 32 字节 base64 备份加密密钥。
8. 后端调用现有 R2 设置保存逻辑写入本地设置文件。
9. 页面刷新 R2 状态并提示配置完成。

## 安全边界

Global API Key 权限过大，只作为一次性 bootstrap 凭据。系统不保存该 key，也不支持从设置页读取它。自动创建的 R2 凭证只允许访问目标 bucket 的对象读写权限。R2 备份默认继续使用 AES-256-GCM 加密；自动配置会生成并保存本地加密密钥，避免配置完成后仍因缺少 `R2_BACKUP_ENCRYPTION_KEY` 不可用。

## 错误处理

Cloudflare 鉴权失败、账号不存在、bucket 创建失败、token 创建失败时，返回可读错误，不保存半成品配置。若 bucket 已创建但 token 创建失败，Cloudflare 侧资源不做删除，因为用户可能指定的是已有 bucket；页面提示失败原因，用户可重试。

## 测试

补充 API 测试覆盖：

- 未登录或缺少 CSRF 时拒绝自动配置。
- 缺少 `BLOG_DATA_ROOT` 时拒绝保存。
- 缺少 Cloudflare 字段时返回 400。
- bucket 已存在时复用并保存 R2 凭证。
- bucket 不存在时创建 bucket 后保存配置。
- Cloudflare 密钥不会写入设置文件或响应。
