# 初始化配置与热更新设计

## 背景

当前项目已经有运行时配置的雏形：

- 编辑器口令可以写入 `data/settings/editor-auth.json`。
- Cloudflare R2 可以写入 `data/settings/cloudflare-r2.json`，并优先于 `.env`。
- Docker 镜像默认设置 `BLOG_DATA_ROOT=/var/lib/blog-navigation`。

现有缺口是：生产环境首次初始化仍依赖环境变量开关或初始化密钥；非 Docker 启动时没有稳定的数据目录默认值；配置读取逻辑分散在认证、R2、数据存储等模块中，热更新边界不统一。

目标是让应用满足三点：

1. 运行配置完全由初始化写入的配置文件承载。
2. 应用不依赖任何外部变量即可启动。
3. 配置保存后可以热更新，不需要重启服务。

## 推荐方案

采用“运行时配置中心 + 首次初始化向导”。

新增一个运行时配置读取层，统一负责：

- 确定默认数据目录。
- 读取 `data/settings/*.json` 配置文件。
- 基于文件 `mtime` 失效缓存。
- 保留环境变量作为兼容输入，但不再作为主配置来源。

首次进入 `/editor/login` 时，如果未检测到编辑口令配置，则显示初始化向导。初始化成功后写入运行时配置文件，并立即创建登录会话。

## 方案权衡

### 方案 A：最小改动

只放宽生产初始化限制，并给 `BLOG_DATA_ROOT` 增加默认值。

优点是改动小、风险低。缺点是配置读取仍然分散，后续 R2、编辑器认证、站点设置继续各自处理环境变量和文件优先级，热更新边界不清晰。

### 方案 B：运行时配置中心

统一引入 `runtime-config`，让认证、R2、站点设置都从同一运行时配置入口读取。

优点是能一次性解决初始化、无变量启动、热更新三件事，也方便后续扩展。缺点是需要小范围重构现有配置读取逻辑，并补充测试覆盖。

### 方案 C：外部配置服务

使用数据库、KV 或对象存储保存配置。

优点是多实例扩展更强。缺点是对当前项目过重，还会引入新的外部依赖，违背“无变量启动”的目标。

本次采用方案 B。

## 配置存储

默认数据目录：

- Docker：继续使用 `/var/lib/blog-navigation`。
- 非 Docker：默认使用项目根目录下的 `data`。
- 如果存在 `BLOG_DATA_ROOT`，仅作为兼容覆盖项。

配置文件：

- `data/settings/editor-auth.json`：编辑器口令哈希和会话状态。
- `data/settings/cloudflare-r2.json`：R2 备份配置和密钥。
- `data/settings/site.json`：站点展示配置。
- `data/manifest.json`：运行时数据资源版本。

环境变量迁移规则：

- 如果运行时配置文件存在，文件优先。
- 如果文件不存在且存在旧环境变量，可以在初始化页或设置页展示“检测到旧配置，可导入”的状态。
- 不自动把敏感环境变量写入文件，避免误持久化管理员未确认的密钥。

## 初始化向导

初始化入口仍使用 `/editor/login`。

未初始化时展示向导：

- 设置编辑口令。
- 确认编辑口令。
- 展示默认数据目录状态和写入权限检测。
- 初始化完成后写入 `editor-auth.json` 并创建登录会话。

安全边界：

- 只有未初始化状态允许执行初始化。
- 已存在 `editor-auth.json` 或 `EDITOR_ACCESS_TOKEN` 时禁止重复初始化。
- 初始化接口保留现有速率限制。
- 生产环境不再强制要求 `EDITOR_RUNTIME_AUTH_SETUP_TOKEN`。
- 如果用户显式配置了 `EDITOR_RUNTIME_AUTH_SETUP_TOKEN`，则初始化向导仍要求输入该密钥，作为可选加固。

## 热更新

运行时配置中心按文件状态做缓存失效：

- 每次读取先检查文件 `mtimeMs` 和 `size`。
- 文件未变化时复用内存缓存。
- 文件变化时重新解析。
- 写入配置时走原子写入，并清理相关缓存。

R2 热更新：

- 设置页保存 `cloudflare-r2.json` 后立即生效。
- S3 客户端缓存键包含 endpoint、accessKeyId、secret hash。
- 配置变化后自动创建新的客户端。

编辑口令热更新：

- 修改或初始化 `editor-auth.json` 后立即影响登录校验。
- 旧会话是否继续有效由配置文件中的 active session 决定。

站点设置热更新：

- 保存 `site.json` 后，通过现有 manifest/revision 机制更新页面数据。

## 错误处理

主要错误类型：

- 数据目录不可写：初始化页明确显示目录路径和权限问题。
- 配置 JSON 损坏：保留当前“配置文件损坏，请修复或删除后重试”的行为。
- R2 配置不完整：设置页显示缺失字段，不阻塞应用启动。
- 旧环境变量存在但未导入：不报错，只提示可在设置页保存为运行时配置。

应用启动不得因为 R2、站点设置或编辑口令未初始化而失败。未初始化只锁定编辑区写操作，不影响公开页面访问。

## 测试策略

单元测试：

- 默认数据目录解析。
- 文件配置优先于环境变量。
- 配置文件 `mtime` 变化后重新读取。
- 未初始化生产环境允许显示初始化向导。
- 已初始化状态禁止重复初始化。

接口测试：

- `GET /api/editor-auth` 在未初始化状态返回 `setupEnabled=true`。
- `PUT /api/editor-auth` 可在无环境变量时初始化。
- `GET /api/data/cloudflare-r2` 正确返回 `source=file/env/default`。
- `PUT /api/data/cloudflare-r2` 保存后下一次读取立即返回新配置。

构建验证：

- `npm run lint -- --fix`
- `npm run typecheck`
- `npm test`

## 成功标准

- 清空编辑器和 R2 相关环境变量后，应用仍能启动。
- 首次访问 `/editor/login` 可以完成初始化。
- 初始化后的编辑口令写入运行时数据目录。
- R2 配置在设置页保存后立即生效。
- 修改运行时配置文件后，无需重启即可被下一次读取使用。
- 公开页面不因未初始化编辑区而不可访问。
