# 内容自动备份与 Docker 更新保留数据设计

## 目标

编辑器保存文章、导航或站点设置后继续触发远端备份；服务运行期间每 3 小时补一次完整远端备份；Docker 更新前先创建本地数据快照，避免镜像更新导致重新配置或数据丢失。

## 方案

1. 保存即备份沿用现有 R2 队列：文章、导航、设置写入接口已经在保存成功后调用 `queueCurrentBackupToRemote`，继续复用现有明文 JSON 上传、重试和 pending 队列。
2. 启动任务增加 3 小时调度：`startServerStartupTasks` 在 Node 服务启动时先 drain 历史 pending 任务，再启动一个单例 interval，每 3 小时入队一次 `scheduled-3h` 备份，写入 latest 和 snapshot。
3. Docker 更新脚本增加更新前本地快照：更新 latest 前把 `/opt/blog-nevigation/.env` 和 `/opt/blog-nevigation/data` 打包到 `/opt/blog-nevigation/backups/`，并保留最近 10 份。

## 取舍

- 复用 R2 队列比新增备份系统更简单，也能保留现有配置和失败重试行为。
- 3 小时备份只有在 R2 已启用且配置有效时才会上传；未配置 R2 时，Docker 持久化数据仍保存在宿主机 `data` 目录。
- 更新前本地快照不替代 R2 远端备份，它主要解决更新镜像前后的误操作和配置丢失风险。

## 验证

- 增加启动任务单测，验证启动会 drain pending 任务，并且 3 小时后只入队一次定时远端备份。
- 运行 lint、typecheck 和相关单测。
