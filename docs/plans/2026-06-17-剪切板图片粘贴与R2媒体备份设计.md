# 剪切板图片粘贴与 R2 媒体备份设计

## 背景

当前文章编辑器只能手动输入图片 URL。用户希望在编辑文章时直接从剪切板粘贴图片，并且图片要随现有 Cloudflare R2 备份链路一起备份，避免 Docker 更新、服务器磁盘故障或远端恢复后文章图片丢失。

现有 R2 自动备份有项目级硬约束：

- R2 自动备份必须保持明文 JSON。
- 不得新增、恢复或要求任何备份加密口令字段。
- R2 `latest/backup.json` 必须能直接 `JSON.parse`。
- R2 自动配置已经生成 bucket 级对象读写凭证，不应再要求用户配置第二套媒体 R2 凭证。

因此图片二进制不能塞进 `backup.json`，应作为独立媒体对象保存和同步。

## 成功标准

- 在文章编辑 textarea 内粘贴剪切板图片后，自动上传图片并插入 Markdown 图片语法。
- 图片文件保存在宿主机持久化数据目录下，Docker 更新不能丢失。
- 图片文件使用现有 R2 配置同步到 R2，不新增 `R2_MEDIA_*` 环境变量或 UI 字段。
- R2 JSON 备份仍是明文 JSON，不包含图片二进制或 base64。
- 从 R2 恢复时能恢复文章数据，并按媒体清单拉回缺失图片。
- R2 未配置或同步失败时，本地文章编辑不被阻断，但 UI/API 返回明确同步状态。

## 方案比较

### 方案 A：base64 内联到 Markdown 或 JSON

优点是实现最快，不需要新增媒体目录或文件服务。

缺点是文章内容和 `backup.json` 会快速膨胀，容易触发现有 5MB R2 备份体积限制；图片无法复用，渲染和迁移也差。该方案还会让 R2 自动备份从“轻量数据快照”变成大文件载荷，不符合当前项目方向。

结论：不采用。

### 方案 B：图片直接上传 R2，文章引用 R2 公网 URL

优点是本地数据目录压力小，未来可以接 CDN。

缺点是要求 R2 配置公网访问域名或 `r2.dev`，而现有自动配置只负责对象读写，不负责公网发布。R2 配置异常时图片链路直接断裂；恢复时也缺少本地完整副本。

结论：不作为第一版主方案。

### 方案 C：本地媒体目录 + R2 媒体对象同步

优点是图片首先落到本地持久化目录，文章引用稳定的站内路径；R2 继续用于远端备份和恢复。图片二进制作为独立对象同步，JSON 备份只记录媒体元数据，符合项目规则。该方案与 EasyMDE、Decap CMS、Joplin、主流 CMS storage adapter 的做法一致：正文存引用，媒体单独存储。

缺点是需要新增媒体清单、媒体文件服务、媒体同步和恢复逻辑。

结论：采用。

## 总体架构

新增媒体层，复用现有编辑器鉴权、CSRF、运行时数据目录和 R2 配置。

本地目录结构：

```text
data/
  media/
    manifest.json
    files/
      2026/
        06/
          <sha256>.png
```

R2 对象结构：

```text
<prefix>/latest/backup.json
<prefix>/snapshots/yyyy/mm/dd/<snapshot>.json
<prefix>/media/files/yyyy/mm/<sha256>.png
```

文章内容只保存站内路径：

```markdown
![图片](/media/files/2026/06/<sha256>.png)
```

R2 JSON 备份可包含媒体清单元数据，但不包含图片二进制。

## 数据模型

新增 `data/media/manifest.json`：

```json
{
  "version": 1,
  "updatedAt": "2026-06-17T00:00:00.000Z",
  "assets": [
    {
      "id": "sha256...",
      "path": "files/2026/06/sha256.png",
      "publicPath": "/media/files/2026/06/sha256.png",
      "mimeType": "image/png",
      "size": 123456,
      "hash": "sha256...",
      "createdAt": "2026-06-17T00:00:00.000Z",
      "updatedAt": "2026-06-17T00:00:00.000Z"
    }
  ]
}
```

字段规则：

- `id` 使用内容 SHA-256，天然去重。
- `path` 是媒体根目录内的相对路径，禁止绝对路径和 `..`。
- `publicPath` 是文章和前端预览使用的路径。
- `mimeType` 只允许 `image/png`、`image/jpeg`、`image/webp`、`image/gif`。
- `hash` 记录文件内容 SHA-256，用于恢复校验。

## 上传流程

前端：

1. `MarkdownEditor` 监听 `onPaste`。
2. 从 `clipboardData.items` 找到第一张或多张 `image/*`。
3. 对每张图片调用 `POST /api/data/media`。
4. 上传过程中在光标位置插入临时占位，避免用户不知道操作状态。
5. 上传成功后替换为真实 Markdown 图片链接。
6. 上传失败后替换或显示错误提示，不吞掉用户剪切板内容。

后端：

1. `POST /api/data/media` 必须通过编辑器写权限和 CSRF。
2. 使用 `multipart/form-data` 接收图片文件。
3. 限制单文件大小，第一版建议 5MB。
4. 用 magic bytes 校验文件类型，不只信任浏览器传入的 MIME。
5. 计算 SHA-256，生成目标路径。
6. 原子写入 `data/media/files/yyyy/mm/<hash>.<ext>`。
7. 更新 `data/media/manifest.json`。
8. 若 R2 已配置，上传同一文件到 `<prefix>/media/<path>`；失败进入媒体同步队列或返回可重试状态。

## 公开访问流程

新增站内媒体访问路径，优先选择：

```text
GET /media/files/yyyy/mm/<hash>.<ext>
```

服务端只允许读取 `data/media/files/` 下的文件，并校验路径不能逃逸媒体目录。返回时设置正确 `Content-Type` 和合理缓存头。

如果 Next.js 路由实现更简单，也可以内部使用：

```text
GET /api/media/files/yyyy/mm/<hash>.<ext>
```

但插入文章的 Markdown 建议保持 `/media/files/...`，后续迁移到静态文件或反代更自然。

## R2 自动配置适配

现有 Cloudflare R2 自动配置已经完成这些工作：

- 创建或确认 bucket。
- 创建 R2 专用 token。
- token 包含 bucket item read/write 权限。
- 保存 bucket、endpoint、access key、secret key、prefix、snapshot 设置。

媒体同步直接复用 `getR2BackupConfig()` 返回的配置，不新增：

- `R2_MEDIA_BUCKET`
- `R2_MEDIA_PREFIX`
- `R2_MEDIA_ACCESS_KEY_ID`
- `R2_MEDIA_SECRET_ACCESS_KEY`
- 任何媒体专用 UI 输入框

同一 bucket 和 prefix 下区分 JSON 备份与媒体对象：

```text
<prefix>/latest/backup.json
<prefix>/snapshots/...
<prefix>/media/files/...
```

这样 `/setup` 和 `/editor/settings` 的现有 R2 自动配置可以无感支持媒体同步。

## 备份流程

现有 R2 JSON 备份流程保持不变：

- `latest/backup.json` 继续是明文 JSON。
- snapshot 继续是明文 JSON。
- 不添加图片 base64。
- 不添加加密口令字段。

需要扩展 `EditorBackupPayload` 的数据部分，使其包含媒体清单元数据：

```json
{
  "data": {
    "articles": [],
    "navigation": [],
    "settings": {},
    "media": {
      "version": 1,
      "assets": []
    }
  }
}
```

兼容性要求：

- 旧备份没有 `media` 时仍可恢复。
- 新备份的 `media` 只表示应恢复的媒体对象列表，不包含文件内容。

## 恢复流程

从 R2 latest 恢复时：

1. 下载 `latest/backup.json`。
2. 校验并恢复文章、导航、设置。
3. 如果 backup payload 中包含媒体清单，则恢复 `data/media/manifest.json`。
4. 遍历媒体清单，检查本地文件是否存在。
5. 文件缺失或 hash 不匹配时，从 `<prefix>/media/<path>` 下载。
6. 下载后校验 hash，成功后写入本地媒体目录。
7. 返回媒体恢复统计：成功、跳过、失败。

部分媒体恢复失败时，不应回滚已恢复的文章数据，但响应必须带 warning，告诉用户哪些图片缺失。

## 错误处理

- 上传非图片：返回 400，提示文件类型不支持。
- 文件过大：返回 413，提示单图大小限制。
- 未登录或 CSRF 失败：沿用现有编辑器鉴权响应。
- 本地写入失败：返回 500，不插入图片链接。
- R2 未配置：本地上传成功，返回 `remoteBackup.enabled = false`。
- R2 上传失败：本地上传成功，返回 `remoteBackup.success = false`，并进入可重试状态。
- 恢复时 R2 媒体对象缺失：记录失败项，恢复接口返回 warning。

## 安全与校验

- 路径必须由服务端生成，禁止用户提交最终存储路径。
- 所有文件读取和写入都必须限制在 `data/media` 内。
- 文件扩展名由 magic bytes 和 MIME 映射得到。
- 响应不返回本地绝对路径。
- 不在审计日志里记录 R2 secret、session token 或完整敏感 header。

## 测试计划

单元测试：

- 媒体 manifest 解析、写入、去重。
- 图片 magic bytes 校验。
- R2 media key 生成，确认不会覆盖 `latest/backup.json`。
- backup payload 包含媒体清单但不包含图片二进制。

API 测试：

- `POST /api/data/media` 成功写入本地文件和 manifest。
- 拒绝未登录、CSRF 缺失、非图片、超大文件。
- R2 未配置时本地上传仍成功。
- R2 已配置时上传媒体对象到 `<prefix>/media/...`。

恢复测试：

- 旧备份没有 media 字段时仍能恢复。
- 新备份能恢复媒体清单。
- 缺失媒体对象返回 warning。
- hash 不匹配时拒绝写入损坏文件。

前端测试：

- textarea 粘贴图片后调用上传接口。
- 成功后插入 Markdown 图片链接。
- 失败后显示明确错误，不破坏已有内容。

## 分阶段实施

第一阶段：媒体基础设施

- 新增 `media` manifest 类型与读写模块。
- 新增本地媒体保存和访问接口。
- 新增上传 API。

第二阶段：编辑器粘贴体验

- `MarkdownEditor` 增加 paste 图片处理。
- 上传成功插入 Markdown。
- 上传状态和错误提示接入现有编辑页状态提示。

第三阶段：R2 媒体同步

- 复用现有 R2 配置上传媒体对象。
- 新增媒体同步结果返回和失败重试。
- 扩展 R2 恢复流程下载媒体对象。

第四阶段：备份 payload 与文档

- backup payload 增加可选 media 清单。
- 更新 API 契约和 README。
- 补全测试，确认 R2 JSON 备份仍可直接 `JSON.parse`。

## 暂不纳入第一版

- 完整媒体库 UI。
- 图片裁剪、压缩和格式转换。
- 孤儿图片自动删除。
- R2 公网 URL 直链。
- 多 bucket 或媒体专用 R2 凭证。
