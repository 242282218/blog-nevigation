# 首次启动引导与全变量前端配置设计

## 背景

当前项目已经有部分运行时配置能力：

- 编辑口令可以通过首次初始化写入 `data/settings/editor-auth.json`。
- Cloudflare R2 可以通过设置页写入 `data/settings/cloudflare-r2.json`。
- 站点设置可以通过设置页写入 `data/settings/site.json`。
- 数据目录已经有默认值，非 Docker 默认使用项目 `data`，Docker 默认使用 `/var/lib/blog-navigation`。

新的目标是：

1. 首次启动必须有清晰的初始化引导。
2. `.env.example` 中的所有变量都能在前端配置。
3. 应用不依赖任何变量也能启动。
4. 能热更新的配置保存后立即生效；不能安全热更新的配置必须明确标记生效方式。

## 推荐方案

采用“首次启动向导 + 运行时配置中心 + 生效等级标记”。

新增统一运行时配置文件：

```text
data/settings/app-runtime.json
```

读取优先级：

1. 前端保存的运行时配置文件。
2. 旧环境变量兼容值。
3. 系统默认值。

环境变量不再作为主配置来源，只用于兼容旧部署、初始化表单默认值和少量构建平台兜底。

## 变量范围

`.env.example` 中的变量全部支持前端配置：

| 变量 | 前端字段 | 存储位置 | 生效等级 |
| --- | --- | --- | --- |
| `EDITOR_ACCESS_TOKEN` | 编辑口令 | `editor-auth.json` | `hot` |
| `BLOG_DATA_ROOT` | 运行时数据目录 | `app-runtime.json` | `restart-required` |
| `COOKIE_SECURE` | Cookie 安全模式 | `app-runtime.json` | `next-session` |
| `TRUSTED_PROXY_IPS` | 可信代理 IP | `app-runtime.json` | `hot` |
| `NEXT_PUBLIC_SITE_URL` | 公开站点 URL | `app-runtime.json` | `hot-server` |
| `R2_BACKUP_ENABLED` | 启用 R2 备份 | `cloudflare-r2.json` | `hot` |
| `R2_ACCOUNT_ID` | Cloudflare Account ID | `cloudflare-r2.json` | `hot` |
| `R2_BUCKET` | R2 Bucket | `cloudflare-r2.json` | `hot` |
| `R2_ACCESS_KEY_ID` | R2 Access Key ID | `cloudflare-r2.json` | `hot` |
| `R2_SECRET_ACCESS_KEY` | R2 Secret Access Key | `cloudflare-r2.json` | `hot` |
| `R2_PREFIX` | R2 对象前缀 | `cloudflare-r2.json` | `hot` |
| `R2_ENDPOINT` | R2 Endpoint | `cloudflare-r2.json` | `hot` |
| `R2_SNAPSHOT_ON_WRITE` | 写入时创建 snapshot | `cloudflare-r2.json` | `hot` |
| `R2_BACKUP_ENCRYPTION_KEY` | R2 备份加密密钥 | `cloudflare-r2.json` | `hot` |
| `R2_ALLOW_PLAINTEXT_BACKUP` | 允许明文备份 | `cloudflare-r2.json` | `hot` |

生效等级含义：

- `hot`：保存后下一次服务端读取立即生效。
- `hot-server`：服务端生成 URL 时立即生效；不能依赖客户端构建期 `NEXT_PUBLIC_*` 注入。
- `next-session`：下一次设置 Cookie 或重新登录后生效。
- `restart-required`：前端可配置，但需要迁移和重启确认，不能无锁热切。

## 首次启动向导

新增 `/setup` 作为独立初始化入口。

未初始化判定：

- 不存在 `data/settings/app-runtime.json`。
- 不存在 `data/settings/editor-auth.json`，且没有兼容的 `EDITOR_ACCESS_TOKEN`。
- 可选：存在旧环境变量但未导入时，仍视为需要完成初始化确认。

引导步骤：

1. **基础配置**
   - 公开站点 URL。
   - Cookie 安全模式。
   - 可信代理 IP。
   - 展示每个字段的当前来源：运行时配置、环境变量、默认值。

2. **数据目录**
   - 展示当前数据目录路径和来源。
   - 检查目录是否可写。
   - 允许填写新目录，但标记为需要迁移和重启。
   - 默认使用当前目录，避免首次启动变成高风险迁移流程。

3. **编辑口令**
   - 设置编辑区登录口令。
   - 写入 `editor-auth.json`，不把明文口令写入 `app-runtime.json`。
   - 成功后立即创建登录会话。

4. **R2 备份**
   - 可跳过。
   - 可手动填写 R2 配置。
   - 可使用现有一键配置流程。
   - 明文备份作为高级危险项，默认关闭。

5. **确认完成**
   - 写入 `app-runtime.json`。
   - 写入或跳过 `cloudflare-r2.json`。
   - 写入 `setupCompletedAt`。
   - 进入 `/editor`。

## 运行时配置模型

`app-runtime.json` 示例：

```json
{
  "version": 1,
  "setupCompletedAt": "2026-06-05T00:00:00.000Z",
  "publicSiteUrl": "https://example.com",
  "cookieSecure": true,
  "trustedProxyIps": ["203.0.113.10"],
  "dataRoot": {
    "path": "/var/lib/blog-navigation",
    "pendingPath": null,
    "requiresRestart": false
  },
  "updatedAt": "2026-06-05T00:00:00.000Z"
}
```

安全规则：

- 敏感密钥不回显完整值。
- 编辑口令只保存哈希。
- R2 密钥保存到 `cloudflare-r2.json`，文件权限尽量收紧到 `0600`。
- `R2_ALLOW_PLAINTEXT_BACKUP` 必须在 UI 中明确标记风险，默认关闭。

## 数据目录切换

`BLOG_DATA_ROOT` 是特殊变量，不能像普通字段一样立即热更新。

原因：

- 配置文件本身也存放在数据目录中。
- 文章、导航、站点设置、备份配置和 manifest 都依赖同一个根目录。
- 热切可能导致旧目录读、新目录写，造成数据分裂。

处理方式：

1. 前端允许配置新目录。
2. 服务端检查新目录可写。
3. 提供迁移动作：复制当前数据到新目录并校验 manifest。
4. 写入 `pendingPath`。
5. 提示用户重启服务后生效。

本期实现可以先支持“配置并提示重启”，迁移动作可作为第二阶段。

## 热更新边界

配置中心按文件状态做缓存失效：

- 每次读取先检查 `mtimeMs` 和 `size`。
- 文件未变化时复用缓存。
- 文件变化时重新解析。
- 写入配置时原子落盘并清理相关缓存。

模块接入：

- `editor-auth-runtime` 读取 `editor-auth.json`，初始化后立即可登录。
- `r2-backup-storage` 读取 `cloudflare-r2.json`，保存后下一次备份或恢复立即生效。
- `site-url` 改为读取运行时 `publicSiteUrl`，只在服务端使用。
- `request-origin` 和 `request-client` 改为读取运行时 `trustedProxyIps`。
- `editor-auth` 的 Cookie options 改为读取运行时 `cookieSecure`。
- `runtime-config` 保留 `BLOG_DATA_ROOT` 兼容覆盖，但新增 pending data root 状态。

## 前端入口

新增或调整页面：

- `/setup`：首次启动向导。
- `/editor/settings/runtime`：运行时配置页，允许后续修改所有变量。
- `/editor/settings`：保留站点设置和 R2 面板，可链接到运行时配置页。

路由行为：

- 未初始化访问 `/editor` 时跳转 `/setup?next=/editor`。
- 未初始化访问 `/editor/login` 时提示进入 `/setup`。
- 公开页面继续使用默认配置，不因未初始化编辑区而失败。

## 错误处理

- 数据目录不可写：显示路径和权限错误，不继续初始化。
- 配置 JSON 损坏：显示修复提示，禁止覆盖敏感配置。
- R2 配置不完整：允许保存为未启用或不完整状态，不阻塞启动。
- 代理 IP 配置无效：拒绝保存，避免生产环境限流绕过。
- Cookie 安全模式变更：提示需要重新登录后完全生效。
- 数据目录变更：提示需要迁移和重启，不承诺立即生效。

## 测试策略

单元测试：

- `app-runtime.json` 解析和默认值。
- 文件配置优先于环境变量。
- `mtime` 或 `size` 变化后重新读取。
- `COOKIE_SECURE` 从运行时配置生成 Cookie options。
- `TRUSTED_PROXY_IPS` 从运行时配置影响请求客户端识别。
- `NEXT_PUBLIC_SITE_URL` 从运行时配置影响服务端 URL。
- `BLOG_DATA_ROOT` 改动只进入 pending 状态，不直接热切。

接口测试：

- `GET /api/setup` 返回初始化状态、字段来源和默认值。
- `PUT /api/setup` 完成首次初始化并创建会话。
- `GET /api/runtime-config` 返回脱敏配置。
- `PUT /api/runtime-config` 保存非敏感配置并立即影响下一次读取。
- R2 设置保存后下一次状态读取来源为 `file`。

界面测试：

- 首次访问编辑区出现初始化引导。
- 向导能完成编辑口令初始化。
- R2 步骤可跳过。
- 高级危险项默认折叠或明确标记。

构建验证：

- `npm run lint -- --fix`
- `npm run typecheck`
- `npm test`

## 成功标准

- 清空 `.env.local` 后应用仍能启动。
- 首次访问后台能进入 `/setup` 并完成初始化。
- `.env.example` 中所有变量都能在前端看到对应配置入口。
- 编辑口令、R2、站点 URL、可信代理 IP 保存后无需重启即可影响下一次服务端读取。
- Cookie 安全模式修改后，下一次登录使用新配置。
- 数据目录修改不会静默热切，必须显示 pending 和重启要求。
- 公开页面不因未初始化后台而不可访问。
