# 架构优化设计方案

> 生成时间：2026-06-06
> 范围：数据流 / 配置流 / 备份恢复流 / API 边界 / 存储层抽象 / 渐进式迁移
> 原则：保持「单容器 + 挂载卷」部署简单性不变；不引入数据库除非收益明确；外科手术式重构，不破坏现有测试语义
> 配套：[项目全量审查报告](./项目全量审查报告.md)、[数据安全与备份恢复优化方案](./数据安全与备份恢复优化方案.md)、[分阶段优化路线图](./分阶段优化路线图.md)

## 一、当前架构问题（按结构性严重度）

### 1.1 硬天花板：单实例强绑定（P3-1）
文件锁（`editor-data-lock.ts:29` 进程内 `heldEditorDataLocks` Map）、本地挂载卷、进程内运行时缓存三者**都假设单进程**。一旦起多副本：
- 文件锁跨容器：mkdir 目录锁能互斥，但重入计数仅单进程可见，stale 清理时序会失配（CC-1）；
- 进程内缓存（2s TTL）各实例不一致，读到陈旧数据（CC-5）；
- 多 worker 共享同一 R2 latest key，并发 PUT last-writer-wins，旧覆盖新（CC-5）。

**结论**：这是当前架构的扩展上限，扩展前必须换存储层。但对「个人/小团队单实例」定位，这是**合理取舍**，不是必须立即修的 bug。

### 1.2 自研持久化层 TCO 高（P3-2）
`editor-data-storage.ts`(1140 行) + `editor-data-lock.ts`(258 行) ≈ 在文件系统上手写了一个迷你数据库（manifest 乐观锁 + hash 完整性 + 原子写 + 文件锁 + 崩溃恢复状态机）。功能正确，但边界 case（磁盘满、锁泄漏、崩溃时序）需要持续测试维护。

### 1.3 双写非事务（P1-10）
`writeXxxToDisk`（数据文件）+ `touchEditorDataResourceManifest`（manifest.json）是**两次独立原子写**，无包裹事务。若 manifest 写失败，数据已更新而 manifest 落后，二者漂移且无补偿回滚（RB-2）。常规读路径又不校验 manifest hash（CR-6），漂移不会被发现。

### 1.4 读路径耦合写锁（CC-4 / DL-4）
`recoverIncompleteRestoreForRead` 在无锁时主动 `acquireEditorDataRootLock`。高并发公开页 SSR 同时触发多次读，争抢写锁，与编辑写入互相 5s 超时，放大延迟。同步读路径遇 restore-state 直接抛错不恢复，SSR 公开页持续 500 直到一次异步写自愈。**读不应该需要写锁。**

### 1.5 读性能 O(n)（P3-4）
运行时模式 `getPosts` 每次读全量 `articles.json` 并 filter（`markdown.ts:471-487`），>1000 篇首页变慢。

### 1.6 缓存一致性窗口（P2-6）
`runtimePostsCache` 5s TTL + `editorDataCache` 2s（mtime+size）。编辑器保存后公开页延迟可见；秒级 mtime FS 下 size 不变时有陈旧读窗口（CR-5）。

### 1.7 API 路由重复样板
`src/app/api/*` 各路由重复 auth 校验 + body 解析 + 错误处理，可抽 HOF 减约 50% 重复。

## 二、推荐目标架构（演进而非推倒）

核心思想：**用一个 `Repository` 接口把存储实现与业务逻辑解耦**，让「文件实现 → SQLite 实现 → 共享后端」可渐进替换，同时**读写路径分离**消除读对写锁的依赖。

```
                       ┌─────────────────────────────────────┐
   公开页 (RSC/SSR) ──▶ │ ReadModel (只读，无锁)               │
                       │  - getPublishedPosts() 走索引/缓存    │
                       │  - 命中 ISR/revalidateTag             │
                       └───────────────┬─────────────────────┘
                                       │ (只读)
   编辑器 API ────────▶ ┌─────────────▼─────────────────────┐
   (withEditorAuth HOF) │ ContentRepository 接口            │
                        │  read / write(revision) / manifest │
                        │  ── FileRepository (当前实现)      │  ← 默认
                        │  ── SqliteRepository (可选/未来)   │
                        └───────────────┬───────────────────┘
                                        │ 写成功事件
                        ┌───────────────▼───────────────────┐
                        │ BackupCoordinator (持久化队列)     │
                        │  落盘 pending → drain → R2(加密)    │
                        │  + revalidateTag 触发读模型失效     │
                        └────────────────────────────────────┘
```

### 2.1 模块边界（目标）

| 层 | 模块 | 职责 | 不允许做 |
|----|------|------|----------|
| 读模型 | `ReadModel`（新，从 markdown.ts 拆出） | 公开页只读，走索引/ISR，**绝不获取写锁** | 写、加锁 |
| 仓储接口 | `ContentRepository`（新接口） | 定义 read/write/manifest 契约 | 绑定具体存储 |
| 文件实现 | `FileRepository`（现 editor-data-storage） | 文件锁 + 原子写 + manifest + restore | 业务规则 |
| 备份协调 | `BackupCoordinator`（增强 editor-remote-backup） | 持久化队列 + 加密 + 触发读模型失效 | 阻塞写响应 |
| API 边界 | `withEditorAuth` HOF（新） | session + origin + CSRF + body 解析 + error boundary | 业务逻辑 |
| 配置 | `app-runtime-config`（现有，保留） | 三层配置 + 热更新 | — |

### 2.2 为什么不直接上数据库
架构对比 agent 结论：当前数据量级（R2 单文件 5MB 限制）+ 单实例下，**原子写 + 目录锁 + manifest 校验已满足一致性**，引入 DB 大幅增加部署运维复杂度，违背 KISS。**目标架构先抽接口、保持文件实现**，仅在出现明确的并发/扩展需求时，才用 SQLite(WAL) 替换——且 SQLite 仍是单文件可挂卷，保留部署简单性。这避免了「为不存在的场景过度设计」。

## 三、关键数据流（目标）

### 3.1 写入流（增强双写事务 + 读模型失效）
```
PUT /api/data/articles
  → withEditorAuth(session+origin+CSRF+parse)        【新 HOF】
  → repo.write(articles, revision)
      在文件锁内：
        ① 比对 revision（冲突 409）
        ② 将 data + manifest 一并写 staging        【新：双写合二为一】
        ③ 回读 staging 校验 hash
        ④ 原子 rename 替换（data 与 manifest 同批）
  → emit WriteSucceeded 事件
      → BackupCoordinator.enqueue（落盘 pending）   【P0-2 持久化队列】
      → revalidateTag(`posts`)                       【P2-6 主动失效】
  → 返回新 revision
```
解决：P1-10（双写事务）、P2-6（缓存失效）、P0-2（持久化队列）。

### 3.2 读取流（无锁 + ISR）
```
/posts/[slug] (RSC)
  → ReadModel.getPost(slug)         无锁
  → 命中 ISR 缓存 or 读索引/articles.json
  → 遇 restore-state：返回上一致缓存或 503（不抢锁）  【DL-4 修复】
```
解决：CC-4、DL-4、P3-4（配合索引）。

### 3.3 配置流（保持，文档化优先级）
`app-runtime.json` > 环境变量 > 默认值（publicSiteUrl/cookieSecure/trustedProxyIps）；R2：`cloudflare-r2.json` > `R2_*` env > disabled；editorAuth：`editor-auth.json` > `EDITOR_ACCESS_TOKEN` > null。
- 优化（CR/config）：`readStoredAppRuntimeConfig` 基于 mtime+size 缓存，原子替换更新下高并发可能读旧缓存 → 缓存键改用 `ino + mtimeNs` 或内容 hash（`app-runtime-config.ts:292-330`）。

### 3.4 备份恢复流（详见数据安全方案）
写后异步 → `BackupCoordinator` 落盘 pending → drain → 明文 JSON 上传 R2（latest + 可选 snapshot）。恢复走 `restoreEditorDataRootAtomically`（staging 校验 → 备份现有[加 fsync] → 状态机 replacing/committed → 崩溃自动 recover）。

### 3.5 API 边界（统一）
```ts
// 目标：src/app/api/_lib/with-editor-auth.ts（新）
export function withEditorAuth<T>(
  handler: (req: NextRequest, body: T) => Promise<Response>,
  schema: ZodSchema<T>,          // 引入 zod 替换 unknown（代码质量项）
): RouteHandler
// 内部：ensureEditorWriteRequest(session+origin+CSRF) → schema.parse → try/catch → 标准错误响应
```
解决：API 重复样板、`unknown` 无 schema 校验（`articles/route.ts:26-29`）。

## 四、可立即落地的架构优化（不破坏现状，P1–P2）

### 4.1 抽取 `withEditorAuth` HOF（P2，开发体验）
- **问题**：`src/app/api/data/{articles,navigation,settings}/route.ts` 等重复 auth+parse+error。
- **动作**：新建 `src/app/api/_lib/with-editor-auth.ts`，逐路由迁移，引入 zod schema。
- **验证**：现有 `tests/app/editor-data-routes.test.ts`（CSRF/origin/revision/oversize/corrupt 全覆盖）不改语义全绿。
- **工作量**：1-1.5 天。

### 4.2 读路径解耦写锁（P1，CC-4/DL-4）
- **动作**：将崩溃恢复仅绑定到「写路径 + 显式后台恢复任务」；读路径检测到 restore-state 时返回上一致缓存或 503，同步读路径触发一次性后台异步恢复而非直接 500/抢锁。
- **涉及**：`editor-data-storage.ts:114-139,225-270`。
- **验证**：新增并发读 + 恢复中读的测试（返回 503 而非 500、不死锁）；现有读写测试不变。
- **工作量**：1-2 天。

### 4.3 data + manifest 单事务（P1，RB-2/CR-6）
- **动作**：`writeXxxToDisk` 时将 data 与 manifest 一并写 staging 再原子替换；读路径校验 manifest hash 失配时自动重建 manifest 并告警。
- **涉及**：`editor-data-storage.ts:869-874,957-962,703,1082-1097`。
- **验证**：注入「manifest 写失败」场景测试，确认数据与 manifest 不漂移或可自愈。
- **工作量**：1.5-2 天。

### 4.4 已发布文章索引（P3-4，读性能）
- **动作**：写入时维护 `published-articles.json` 索引（slug → meta），`getPosts` 读索引避免全量 filter；或按月份分片 `articles-YYYY-MM.json`，manifest 记录分片，写入只锁当前月。
- **涉及**：`markdown.ts:471-487`、`editor-data-storage.ts`。
- **权衡**：索引方案改动小、收益直接；分片方案支持 10 万+ 文章但复杂度高，**仅在文章量预期 >1000 时启用**，否则属过度设计。
- **验证**：大库（生成 2000 篇 fixture）首页加载基准测试。

### 4.5 缓存一致性（P2-6，CR-5）
- **动作**：写入成功后主动 `revalidateTag`/`resetRuntimePostsCache`（事件驱动失效），缓存键改 `ino+mtimeNs` 或内容 hash。
- **涉及**：`markdown.ts:124-149`、`editor-data-storage.ts:141-162`。
- **验证**：保存后公开页在限定延迟内更新；秒级 mtime FS 下不读陈旧。

## 五、渐进式迁移方案（4 阶段，源自架构对比 agent）

> 每阶段都可独立交付、独立验证、可回退。

### 阶段一：消除文档漂移 + 补齐验证（0.5–1 天，风险低）
- **目标**：先解决表象与维护负担，降低误导风险。
- **动作**：合并 `docs/` 三份审计为单一 ADR + CHANGELOG，标注已修项（CSRF 已在 `851e471` 修复）；对 AUDIT 中 P2 项（原子写、限流、Markdown XSS）补单测固化现状。
- **验证**：`npm run check` 通过；架构测试与新增安全测试全绿。

### 阶段二：存储层抽象与去自研化（2–4 天，风险中）
- **目标**：为可维护性与未来扩展拆开存储耦合。
- **动作**：在 `editor-data-storage` 之上定义 `ContentRepository` 接口（read/write/manifest），**先保持文件实现**；评估用 SQLite(WAL) 替换 manifest+hash+文件锁，保留单文件挂卷形态。
- **验证**：现有读写/备份/恢复测试不改语义全通过；并发写入压力测试无数据损坏。
- **回退**：接口默认仍指向 FileRepository，SQLite 仅作可选实现。

### 阶段三：读路径无状态化（2–3 天，风险中）
- **目标**：解除进程内缓存与单实例耦合，提升读性能。
- **动作**：公开页改 ISR/按需 `revalidate`，写操作触发 `revalidateTag`/`revalidatePath`；移除或外置 2s 进程内缓存。
- **验证**：编辑后公开页在限定延迟内更新；多次请求命中缓存且数据一致。

### 阶段四：多副本与协作能力（按需，3–6 天，风险高）
- **目标**：突破单实例天花板。
- **动作**：存储切共享后端（SQLite 同卷单写或托管 Postgres），锁改后端事务/行锁；鉴权从单口令演进为多用户 + 角色 + 操作审计。
- **验证**：双副本并发写无冲突；多用户权限与审计日志端到端测试通过。
- **前置**：仅当出现真实多人协作/多副本需求时启动，否则停在阶段三。

## 六、可观测性补强（P3-5，贯穿各阶段）
- **问题**：关键操作（写入/备份/restore）只有 `console.error`，无 traceId/耗时/数据量/错误栈。
- **动作**：引入 pino 结构化日志，记录 traceId + 操作 + 耗时 + 数据量；关键指标（写入延迟、锁等待、备份成功率、缓存命中）暴露 `/metrics`（Prometheus）或至少结构化日志可聚合；集成 Sentry 错误上报。
- **权衡**：博客导航站规模有限，**全套 OTel+Prometheus+Grafana 运维成本高**（docker_cicd agent 判定 not_applicable），先用结构化日志 + 现有 webhook 通知，指标按需。

## 七、不建议照搬的模式（避免过度工程）

| 模式 | 不建议原因 | 替代 |
|------|-----------|------|
| 新项目照搬「FS 上自研 manifest+hash+文件锁迷你数据库」 | 并发/崩溃/锁泄漏边界需大量测试维护，无法跨进程 | 有并发预期直接用 SQLite(WAL) 自带原子性与并发读 |
| 进程内 2s 缓存作通用读加速 | 与单实例假设耦合，多副本缓存不一致 | 无状态读 + ISR/CDN 或外置 Redis |
| GitHub 加密备份作主备份通道 | 把 git 当数据后端是 workaround，仓库膨胀、无生命周期 | 对象存储（R2/S3）+ 版本化/生命周期（本项目已有 R2） |
| 公开读强制走磁盘异步读 | 放弃 SSG/CDN 零运行时与强缓存，仅为支持在线编辑 | 写入触发 ISR/按需再生，读回到静态/CDN |
| 为单用户场景引入 Redux/CRDT/实时协作 | 数据规模小并发低，过早优化 | 现有 hook + 乐观锁已足够 |

## 八、成功标准
- 存储实现可通过 `ContentRepository` 接口替换，业务代码零改动；
- 公开页读路径不再获取写锁，恢复中读返回 503 而非 500 死锁；
- data 与 manifest 不再漂移（注入故障测试通过）；
- 保存后公开页延迟可见 ≤ 设定阈值；
- 全程 `npm run check` 通过，现有测试语义不变。
