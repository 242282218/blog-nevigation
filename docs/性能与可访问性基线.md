# 性能与可访问性基线

更新时间：2026-06-08

## 目标

为公开页面建立可重复的 Lighthouse 基线，先收集报告和警告，后续根据真实 CI 数据把稳定阈值升级为失败门禁。

## 覆盖页面

Lighthouse CI 当前覆盖：

```text
/
/blog
/navigation
/posts/2026-05-25-getting-started
```

配置文件：`lighthouserc.cjs`

CI 工作流：`.github/workflows/ui-smoke.yml`

## 当前阈值

当前阈值为 warning，不阻断发布：

| 指标 | 阈值 |
|------|------|
| Performance | >= 0.90 |
| CLS | <= 0.1 |
| LCP | <= 2500ms |
| TBT | <= 200ms |

## 报告产物

CI 会把报告写入：

```text
output/lighthouse
```

并随 UI smoke artifacts 上传。

## 本地运行

先构建并启动服务，再运行：

```bash
npx --yes @lhci/cli@0.15.1 autorun --config=./lighthouserc.cjs
```

本地服务需要监听：

```text
http://127.0.0.1:3210
```

## 可访问性检查原则

当前已有：

- 语义化页面结构和跳转主内容链接。
- 编辑器表单字段错误提示和 `aria-describedby`。
- Playwright/Python UI smoke 覆盖公开页和编辑器主流程。

下一步建议：

1. 在 UI smoke 中加入 axe 检查，先覆盖公开页。
2. 对编辑器的复杂表单保留人工校验清单，避免自动规则遗漏键盘流程问题。
3. 发现稳定问题后写入组件级测试，而不是只依赖截图。

## 升级门禁条件

满足以下条件后，可把 warning 改为 error：

- 连续 10 次主分支 CI 数据稳定。
- 页面内容和测试数据固定。
- 确认 CI runner 性能波动不会频繁误报。
- 已为图片尺寸、懒加载、代码高亮等主要波动源建立约束。
