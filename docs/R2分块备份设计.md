# R2 分块备份设计

## 1. 背景

当前 R2 备份采用单个全量 JSON：

```text
{prefix}/latest/backup.json
```

这个格式简单、透明、可直接 `JSON.parse`，当前真实备份体积很小，恢复已经验证可用。但随着文章和图片数量增长，单个全量 JSON 会出现几个结构性问题：

- 每次备份都上传全量数据，改一篇文章也要重传全部文章。
- 恢复时需要一次性下载、解析、校验整个 JSON。
- 当前代码对 R2 latest JSON 下载有 128MB 上限。
- R2 请求有 30s timeout，大 JSON 更容易受网络抖动影响。
- 图片如果内联进 JSON，会让 JSON 快速膨胀。
- 同一个 `latest/backup.json` 高频覆盖写会增加失败和限流风险。

因此 v2 的目标不是替换 R2，而是让 R2 成为更稳定的长期备份仓库：文章一篇一个对象，图片一张一个对象，快照只保存索引。

## 2. 设计目标

必须满足：

- R2 仍然是优先备份目标。
- R2 结构化备份对象必须保持明文 JSON。
- 不新增、不恢复、不要求任何 R2 加密口令字段。
- 不新增任何 R2 加密相关环境变量、UI 输入框、API 字段或文档说明。
- 结构化备份对象必须能直接 `JSON.parse`。
- 图片不写入文章 JSON，不写入 snapshot JSON。
- 旧 v1 备份必须还能恢复。
- v2 上传失败不能破坏旧可用快照。
- v2 恢复必须 all-or-nothing，不能半恢复。
- Docker 更新和部署不能删除或覆盖 `/opt/blog-nevigation/data` 与生产 `.env`。

非目标：

- 第一阶段不重构本地运行时数据结构。本地仍可继续使用 `articles/articles.json`。
- 第一阶段不实现数据库化。
- 第一阶段不实现复杂 GC 删除；只提供保留策略和后续 GC 方案。
- 第一阶段不引入压缩格式，因为自动 R2 备份链路要求 JSON 可直接解析。

## 3. 参考项目原则

本设计参考成熟备份系统的通用原则，而不是照搬其复杂度：

- WordPress / Ghost / Strapi：内容数据和上传文件分开备份，恢复时必须同时恢复两类资源。
- restic / Borg / Kopia：快照 manifest 指向不可变内容对象，通过 hash 做去重和完整性校验。
- Velero：对象存储作为备份仓库，恢复前先验证备份集完整性。

落到本项目：

- 文章是结构化 JSON 对象。
- 导航、设置、媒体 manifest 是结构化 JSON 对象。
- 图片是原始二进制对象，由 JSON manifest 引用。
- 快照只保存引用和校验信息。
- latest 只保存最新快照指针。

## 4. 总体架构

v2 采用内容寻址对象仓库：

```text
当前本地数据
  -> 规范化 JSON
  -> sha256
  -> 上传不可变对象
  -> 生成 snapshot manifest
  -> 最后更新 latest pointer
```

恢复流程反向执行：

```text
latest pointer
  -> snapshot manifest
  -> article indexes / media indexes
  -> article objects / media objects / settings / navigation
  -> 校验 sha256 + size + schema
  -> 组装现有 EditorBackupData
  -> 调用现有原子恢复
```

关键提交规则：

```text
latest pointer 永远最后写。
```

只要 latest 没有更新，恢复端仍然读取上一次完整快照。因此上传中途失败不会污染可恢复状态。

## 5. R2 Key 结构

在现有 prefix 下新增 `v2/`，不删除 v1：

```text
{prefix}/
  latest/
    backup.json

  v2/
    repo.json
    latest.json

    snapshots/
      2026/
        06/
          27/
            20260627T120000Z-a1b2c3/
              manifest.json
              articles-0000.json
              articles-0001.json
              media-0000.json

    objects/
      articles/
        {articleId}/
          {articleHash}.json
      navigation/
        {navigationHash}.json
      settings/
        {settingsHash}.json
      media-manifests/
        {mediaManifestHash}.json

    media/
      ab/
        {imageHash}.webp
      c1/
        {imageHash}.png
```

说明：

- `latest/backup.json` 是旧 v1，保留兼容。
- `v2/latest.json` 是最新快照指针，体积应长期小于 10KB。
- `v2/snapshots/.../manifest.json` 是一次完整快照的入口。
- `v2/snapshots/.../articles-xxxx.json` 是文章索引分页。
- `v2/snapshots/.../media-xxxx.json` 是媒体索引分页。
- `v2/objects/**` 是不可变 JSON 内容对象。
- `v2/media/**` 是图片二进制对象，不是 JSON 备份壳。

## 6. JSON 格式

所有结构化 JSON 对象统一包含：

```json
{
  "schemaVersion": 2,
  "kind": "..."
}
```

### 6.1 repo.json

用于识别 v2 仓库格式：

```json
{
  "schemaVersion": 2,
  "kind": "repository",
  "name": "blog-navigation-r2-backup",
  "createdAt": "2026-06-27T12:00:00.000Z",
  "format": "chunked-json",
  "features": {
    "articleObjects": true,
    "mediaObjects": true,
    "contentAddressed": true
  }
}
```

创建规则：

- 首次 v2 sync 时如果不存在则创建。
- 已存在时只校验格式，不覆盖不必要字段。

### 6.2 latest.json

最新快照指针：

```json
{
  "schemaVersion": 2,
  "kind": "latest-pointer",
  "updatedAt": "2026-06-27T12:00:00.000Z",
  "snapshotId": "20260627T120000Z-a1b2c3",
  "snapshotKey": "v2/snapshots/2026/06/27/20260627T120000Z-a1b2c3/manifest.json",
  "snapshotHash": "f4c6..."
}
```

校验规则：

- `schemaVersion` 必须是 `2`。
- `kind` 必须是 `latest-pointer`。
- `snapshotKey` 必须在当前 prefix 的 `v2/snapshots/` 下。
- `snapshotHash` 必须等于 manifest 正文的 sha256。

### 6.3 snapshot manifest

一次快照的总清单：

```json
{
  "schemaVersion": 2,
  "kind": "snapshot",
  "snapshotId": "20260627T120000Z-a1b2c3",
  "createdAt": "2026-06-27T12:00:00.000Z",
  "reason": "manual-sync",
  "source": "r2",
  "persistent": true,
  "dataManifest": {},
  "counts": {
    "articles": 5000,
    "categories": 12,
    "media": 300
  },
  "articleIndexes": [
    {
      "key": "v2/snapshots/2026/06/27/20260627T120000Z-a1b2c3/articles-0000.json",
      "sha256": "6b1a...",
      "size": 120000,
      "count": 1000
    }
  ],
  "mediaIndexes": [
    {
      "key": "v2/snapshots/2026/06/27/20260627T120000Z-a1b2c3/media-0000.json",
      "sha256": "9e2d...",
      "size": 80000,
      "count": 300
    }
  ],
  "navigation": {
    "key": "v2/objects/navigation/53aa...json",
    "sha256": "53aa...",
    "size": 20000
  },
  "settings": {
    "key": "v2/objects/settings/12bc...json",
    "sha256": "12bc...",
    "size": 2000
  },
  "mediaManifest": {
    "key": "v2/objects/media-manifests/a871...json",
    "sha256": "a871...",
    "size": 30000
  }
}
```

说明：

- `dataManifest` 保存现有 `EditorDataManifest`，用于恢复前置条件和恢复后校验。
- `counts` 用于 UI 展示和验收。
- `articleIndexes` 和 `mediaIndexes` 是分页索引，不直接保存全文。

### 6.4 article index

文章索引分页：

```json
{
  "schemaVersion": 2,
  "kind": "article-index",
  "snapshotId": "20260627T120000Z-a1b2c3",
  "page": 0,
  "items": [
    {
      "id": "post-1",
      "key": "v2/objects/articles/post-1/6ff2...json",
      "sha256": "6ff2...",
      "size": 12345
    }
  ]
}
```

分页规则：

- 每页最多 1000 篇文章。
- 或者单页 JSON 超过 1MB 时提前切页。
- `items` 按文章 id 或现有顺序稳定排序，必须保证同一份数据生成稳定索引。

### 6.5 article object

单篇文章对象：

```json
{
  "schemaVersion": 2,
  "kind": "article",
  "id": "post-1",
  "data": {}
}
```

规则：

- `data` 是现有 Article 对象。
- hash 基于规范化后的整个 article object。
- 不额外内联图片二进制。
- 不从 Markdown 中临时解析图片作为唯一来源，媒体完整性以 media manifest 为准。

### 6.6 navigation object

```json
{
  "schemaVersion": 2,
  "kind": "navigation",
  "data": []
}
```

`data` 是现有 Category 数组。

### 6.7 settings object

```json
{
  "schemaVersion": 2,
  "kind": "settings",
  "data": {}
}
```

`data` 是现有 SiteSettings。

### 6.8 media manifest object

```json
{
  "schemaVersion": 2,
  "kind": "media-manifest",
  "data": {
    "version": 1,
    "updatedAt": "2026-06-27T12:00:00.000Z",
    "assets": []
  }
}
```

`data` 是现有 EditorMediaManifest。

### 6.9 media index

媒体索引分页：

```json
{
  "schemaVersion": 2,
  "kind": "media-index",
  "snapshotId": "20260627T120000Z-a1b2c3",
  "page": 0,
  "items": [
    {
      "path": "files/2026/06/abc.webp",
      "publicPath": "/media/files/2026/06/abc.webp",
      "key": "v2/media/ab/abc.webp",
      "sha256": "abc...",
      "size": 123456,
      "mimeType": "image/webp"
    }
  ]
}
```

规则：

- `sha256` 必须等于现有 `EditorMediaAsset.hash`。
- `key` 使用 hash 前两位分桶。
- 单图仍遵守当前 5MB 限制。
- 同一张图片多篇文章引用，只上传一次。

## 7. 规范化与 hash

必须实现稳定 JSON 序列化：

```text
stableJson(value) -> string
sha256(stableJson(value))
```

要求：

- object key 按字典序排序。
- array 保留原始顺序。
- 输出不依赖 Node 版本对象插入顺序。
- hash 用 UTF-8 文本计算。
- 上传正文使用同一份 stable JSON，避免“hash 和上传内容不是同一字节”的问题。

建议新增：

```text
src/lib/stable-json.ts
```

导出：

```ts
export function stableJsonStringify(value: unknown): string;
export function sha256Hex(bytesOrText: string | Uint8Array): string;
```

如果项目已有等价内部函数，优先抽出来复用，避免重复实现。

## 8. 上传流程

### 8.1 入口

现有 `syncCurrentBackupToRemote` 保持 API 不变，但内部改为 v2 优先：

```text
syncCurrentBackupToRemote(options)
  -> createCurrentEditorRemoteBackupPackage()
  -> uploadChunkedBackupToR2(package, options)
  -> best-effort upload v1 latest/snapshot
```

第一阶段建议：

- v2 成功即认为 R2 sync 成功。
- v1 作为兼容 best-effort，可记录 warning。
- 如果 v2 失败，不允许写 v2 latest。

### 8.2 详细步骤

1. 获取 R2 配置。
2. 读取当前文章、导航、设置、媒体 manifest、媒体文件。
3. 确保 `v2/repo.json` 存在。
4. 为每篇文章生成 article object：
   - 构造 `{ schemaVersion: 2, kind: "article", id, data }`。
   - stable JSON。
   - sha256。
   - key = `v2/objects/articles/{safeArticleId}/{hash}.json`。
5. 为 navigation/settings/media manifest 生成对象和 key。
6. 为每张图片生成 media key：
   - key = `v2/media/{hash.slice(0, 2)}/{hash}.{ext}`。
   - ext 从 MIME 推导。
7. 上传 immutable 对象。
8. 生成并上传 article index pages。
9. 生成并上传 media index pages。
10. 生成 snapshot manifest，计算 hash，上传。
11. 生成 latest pointer，上传 `v2/latest.json`。
12. 返回上传结果和统计。

### 8.3 上传缓存

新增本地缓存：

```text
data/.r2-v2-upload-cache.json
```

格式：

```json
{
  "version": 1,
  "objects": {
    "v2/objects/articles/post-1/hash.json": {
      "sha256": "hash",
      "size": 12345,
      "uploadedAt": "2026-06-27T12:00:00.000Z"
    }
  }
}
```

规则：

- 缓存命中且 sha256/size 相同，可跳过 PUT。
- 缓存缺失时可直接 PUT，不强制 HEAD。
- PUT 成功后写缓存。
- 缓存写入必须原子写。
- 缓存损坏时忽略并重建，不影响备份正确性。

### 8.4 并发限制

上传并发建议：

```text
JSON 对象 PUT 并发：5
媒体 PUT 并发：3
latest PUT 并发：1
```

理由：

- 避免图片上传占满请求。
- 避免 R2 同 key 高频写入。
- latest 是提交点，必须串行。

### 8.5 latest 防抖

当前自动备份可能由编辑操作触发。为了避免频繁覆盖 `v2/latest.json`：

- 手动同步：立即写 latest。
- 自动同步：5-15 秒内合并多次触发，只写一次 latest。
- 队列里相同原因的 pending sync 可合并。

第一阶段可以先实现最小防抖：

```text
如果 activeDrain 已存在，只入队一次 pending latest sync。
```

## 9. 恢复流程

### 9.1 入口

现有远程恢复接口保持不变：

```text
POST /api/data/backup/remote
{ "action": "restore", "currentManifest": ... }
```

内部逻辑：

```text
try restore v2
if v2 latest not found -> restore v1
if v2 exists but校验失败 -> 失败，不 fallback v1
```

原因：

- v2 存在代表用户已进入新格式。
- v2 校验失败时 fallback v1 可能恢复更旧数据，风险更高。

### 9.2 详细步骤

1. 读取 `v2/latest.json`。
2. 校验 latest pointer 格式。
3. 下载 `snapshotKey`。
4. 校验 snapshot hash。
5. 校验 snapshot manifest 格式。
6. 校验 `currentManifest` 前置条件。
7. 下载并校验 article index pages。
8. 下载每个 article object：
   - 校验 size。
   - 校验 sha256。
   - 校验 schemaVersion/kind/id。
   - 解析为 Article。
9. 下载 navigation object：
   - 校验 hash 和格式。
   - 解析为 Category[]。
10. 下载 settings object：
   - 校验 hash 和格式。
   - 解析为 SiteSettings。
11. 下载 media manifest object：
   - 校验 hash 和格式。
   - 解析为 EditorMediaManifest。
12. 下载 media index pages。
13. 下载每张图片：
   - 校验 ContentLength 不超过 5MB。
   - 校验 size。
   - 校验 sha256。
   - 校验 mimeType。
14. 组装为现有 `EditorBackupData`。
15. 调用 `restoreEditorBackupData` 或底层 `restoreEditorDataRootAtomically`。
16. 恢复成功后触发一次 v2 remote sync，生成恢复后的快照。

### 9.3 all-or-nothing

以下任一情况必须失败，且不能修改本地数据：

- latest pointer 不是有效 JSON。
- snapshot manifest 缺失。
- snapshot hash 不匹配。
- 任一 article index 缺失或 hash 不匹配。
- 任一 article object 缺失或 hash 不匹配。
- navigation/settings/media manifest 缺失或 hash 不匹配。
- 任一 media object 缺失或 hash 不匹配。
- schemaVersion/kind 不匹配。
- currentManifest 前置条件冲突。

恢复只允许在所有远端对象下载和校验完成后进入本地 staging。

## 10. v1 兼容策略

### 10.1 上传兼容

第一阶段：

- v2 是主成功路径。
- v1 上传可以保留，用于旧版本回滚或手工检查。
- v1 上传失败不影响 v2 成功，但返回 warning。

第二阶段：

- 当 v2 稳定后，可配置是否继续写 v1。
- 默认仍建议保留 v1 latest 一段时间，方便人工检查。

### 10.2 恢复兼容

恢复优先级：

```text
v2/latest.json 存在 -> 恢复 v2
v2/latest.json 不存在 -> 恢复 v1 latest/backup.json
```

不做：

```text
v2 存在但损坏 -> 自动 fallback v1
```

这会掩盖 v2 备份损坏问题，也可能恢复旧数据。

## 11. 保留策略和 GC

第一阶段只做保留策略建议，不自动删除对象：

```text
保留最近 30 个快照
保留最近 30 天每日最后一个快照
保留最近 12 个月每月最后一个快照
```

如果要自动删除，只能删除 snapshot manifest 和 index；不要删除 `objects/` 与 `media/`。

第二阶段再做 GC：

1. 列出所有保留快照。
2. 下载保留快照 manifest 和 indexes。
3. 汇总仍被引用的对象 key。
4. 列出 `v2/objects/` 和 `v2/media/`。
5. 找出超过 7 天且无人引用的对象。
6. dry-run 输出报告。
7. 用户确认或配置开启后再删除。

GC 删除必须永远不能影响 latest 指向的快照。

## 12. 错误处理

新增错误类型建议：

```ts
class R2ChunkedBackupNotFoundError extends Error {}
class R2ChunkedBackupFormatError extends Error {}
class R2ChunkedBackupIntegrityError extends Error {}
class R2ChunkedBackupObjectMissingError extends Error {}
class R2ChunkedBackupUploadError extends Error {}
```

错误信息必须包含阶段：

```text
read-latest
read-snapshot
read-article-index
read-article-object
read-media-index
read-media-object
upload-article-object
upload-media-object
write-snapshot
write-latest
```

API 响应建议：

```json
{
  "success": false,
  "error": {
    "code": "r2_chunk_integrity_error",
    "stage": "read-media-object",
    "message": "Media object hash mismatch.",
    "key": "v2/media/ab/abc.webp"
  }
}
```

不要把 R2 secret、access key、Global API Key 写入日志或响应。

## 13. UI 和状态展示

设置页 R2 面板新增展示字段：

```text
备份格式：v2 分块备份
最新快照：20260627T120000Z-a1b2c3
文章对象：5000
图片对象：300
latest 大小：约 1KB
上次同步：2026-06-27 20:00
队列状态：idle / pending / failed
```

恢复前确认文案应明确：

```text
将从 R2 v2 分块快照恢复。恢复会先校验所有文章和图片，校验通过后一次性替换本地数据。
```

失败时展示：

```text
恢复失败：图片对象校验失败，未修改本地数据。
```

不要要求用户输入加密口令。

### 16.2 灰度步骤

1. 本地跑单元测试和集成测试。

4. 用测试 bucket 或真实 R2 镜像前缀跑 v2 sync。
5. 清空测试实例数据。

9. 确认 v1 fallback 仍可用。
10. 再部署生产。

### 16.3 回滚策略

代码回滚：

- v1 `latest/backup.json` 仍保留，旧版本可继续读取 v1。
- v2 数据留在 R2，不影响旧版本。

数据回滚：

- 如果新版本写入错误，不更新 latest 的情况下不会影响 v2 恢复入口。
- 如果 latest 指向错误快照，可手动把 `v2/latest.json` 改回上一份 snapshot pointer。
- 不直接删除 v2 objects，避免误伤历史快照。

## 17. 验收标准

## 19. 风险和对应措施

| 风险 | 影响 | 措施 |
|---|---|---|
| R2 对象数量变多 | list/GC 成本上升 | 第一阶段不依赖 list 恢复，只按 manifest 读取 |
| latest 高频写 | 同 key 写入失败或限流 | 防抖、队列合并、latest 串行写 |
| 上传缓存损坏 | 重复上传或跳过错误 | 缓存只优化性能，不作为正确性依据 |
| v2 manifest 写错 | 恢复失败 | latest 最后写，manifest hash 校验 |
| 图片缺失 | 恢复失败 | all-or-nothing，错误指向具体 key |
| 老版本不识别 v2 | 回滚困难 | 保留 v1 latest |
| 本地 articles.json 仍变大 | 运行时性能问题 | v2 先解决备份；未来再拆本地存储或上 SQLite |

## 20. 最终建议

按方案 A 落地：文章一篇一个 JSON，图片一张一个对象，snapshot manifest 负责把它们串起来。

这条路线最符合当前项目：

- 不改用户使用方式。
- 不破坏已有 v1 备份。
- 不引入数据库复杂度。
- 不违反 R2 明文 JSON 规则。
- 能真实解决数据变大后 latest 过大、恢复超时、全量重传的问题。
