# 代码审查方案

## 审查范围

基于以下两份设计文档的实现：

1. [初始化配置与热更新设计](plans/2026-06-05-初始化配置与热更新设计.md)
2. [首次启动引导与全变量前端配置设计](plans/2026-06-05-首次启动引导与全变量前端配置设计.md)

审查目标：验证运行时配置中心、首次启动向导和热更新机制的实现质量，确保满足设计要求和架构规范。

---

## 审查维度

### 1. 架构一致性

#### 1.1 运行时配置中心

**检查点：**

- [ ] 是否新增了统一的运行时配置读取层
- [ ] 配置文件位置是否符合设计：
  - `data/settings/app-runtime.json`
  - `data/settings/editor-auth.json`
  - `data/settings/cloudflare-r2.json`
  - `data/settings/site.json`
- [ ] 配置优先级是否正确：文件配置 > 环境变量 > 系统默认值
- [ ] 环境变量是否仅用于兼容，不作为主配置来源
- [ ] 默认数据目录是否正确：
  - Docker: `/var/lib/blog-navigation`
  - 非 Docker: `${项目根}/data`

**审查文件：**

- `src/lib/app-runtime-config.ts`
- `src/lib/runtime-config.ts`
- `src/lib/editor-auth-runtime.ts`

**验证方法：**

```bash
# 读取配置逻辑的单元测试
npm run test -- runtime-config
npm run test -- app-runtime-config

# 查看配置优先级处理代码
grep -r "BLOG_DATA_ROOT" src/lib/
grep -r "app-runtime.json" src/lib/
```

#### 1.2 首次启动向导

**检查点：**

- [ ] `/setup` 路由是否实现
- [ ] 未初始化判定逻辑是否正确：
  - 不存在 `app-runtime.json`
  - 不存在 `editor-auth.json` 且无 `EDITOR_ACCESS_TOKEN`
- [ ] 访问 `/editor` 时未初始化是否跳转到 `/setup`
- [ ] 向导步骤是否完整：
  1. 基础配置（站点 URL、Cookie 安全模式、可信代理 IP）
  2. 数据目录（显示路径和权限检查）
  3. 编辑口令（哈希存储，创建会话）
  4. R2 备份（可跳过）
  5. 确认完成（写入配置文件）

**审查文件：**

- `src/app/setup/page.tsx`
- `src/app/setup/SetupWizard.tsx`
- `src/app/api/setup/route.ts`
- `src/lib/setup-state.ts`
- `src/middleware.ts`（路由拦截逻辑）

**验证方法：**

```bash
# 首次启动向导测试
npm run test -- setup

# 界面测试
npm run smoke:editor:login

# 手动验证：清空数据目录后访问
rm -rf data/settings/*.json
npm run dev
# 访问 http://localhost:3000/editor 应跳转到 /setup
```

#### 1.3 热更新机制

**检查点：**

- [ ] 配置缓存是否基于文件 `mtimeMs` 和 `size` 失效
- [ ] 文件变化后是否自动重新解析
- [ ] 写入配置时是否原子操作
- [ ] 写入后是否清理相关缓存
- [ ] 各配置项生效等级是否符合设计：
  - `hot`: 编辑口令、R2、可信代理 IP
  - `hot-server`: 站点 URL
  - `next-session`: Cookie 安全模式
  - `restart-required`: 数据目录

**审查文件：**

- `src/lib/editor-runtime-cache.ts`
- `src/lib/editor-auth-runtime.ts`
- `src/lib/r2-backup-storage.ts`
- `src/lib/request-origin-runtime.ts`

**验证方法：**

```bash
# 热更新测试
npm run test -- editor-runtime-cache
npm run test -- hot-reload

# 手动验证：修改配置文件后立即生效
# 1. 启动服务
npm run dev

# 2. 修改 data/settings/app-runtime.json
# 3. 下一次请求应使用新配置（无需重启）
```

### 3. 安全性

#### 3.1 敏感数据保护

**检查点：**

- [ ] 编辑口令是否只保存哈希，不存明文
- [ ] R2 密钥是否存储在 `cloudflare-r2.json` 而非 `app-runtime.json`
- [ ] 配置文件权限是否设置为 `0600`
- [ ] API 返回配置时是否脱敏敏感字段
- [ ] `R2_SECRET_ACCESS_KEY` 是否不回显完整值
- [ ] 明文备份选项 `R2_ALLOW_PLAINTEXT_BACKUP` 是否默认关闭并明确标记风险

**审查代码：**

```typescript
// 编辑口令存储检查
grep -r "EDITOR_ACCESS_TOKEN" src/lib/editor-auth-runtime.ts
// 应该看到哈希处理，不应该看到明文存储

// R2 密钥存储检查
grep -r "R2_SECRET_ACCESS_KEY" src/lib/
// 应该只在 cloudflare-r2.json 相关逻辑中出现

// API 脱敏检查
grep -r "secretAccessKey" src/app/api/
// 响应中应该脱敏或截断
```

#### 3.2 访问控制

**检查点：**

- [ ] 初始化接口是否保留速率限制
- [ ] 已初始化状态是否拒绝重复初始化
- [ ] 配置修改接口是否需要认证
- [ ] 公开页面是否不因未初始化而不可访问
- [ ] `/setup` 路由在已初始化后是否仍可访问（可能需要禁止）

**审查文件：**

- `src/app/api/setup/route.ts`
- `src/app/api/runtime-config/route.ts`
- `src/middleware.ts`

#### 3.3 输入验证

**检查点：**

- [ ] 站点 URL 是否验证格式
- [ ] 代理 IP 是否验证格式
- [ ] 数据目录路径是否检查可写权限
- [ ] 配置值是否有长度和类型限制
- [ ] JSON 解析是否有异常处理

#### 4.2 错误处理

**检查点：**

- [ ] 配置文件不存在时是否优雅降级
- [ ] JSON 解析失败是否有明确错误信息
- [ ] 数据目录不可写是否明确提示路径和权限问题
- [ ] R2 配置不完整是否不阻塞启动
- [ ] 网络错误、文件系统错误是否有重试或降级机制
- [ ] Go 风格错误是否带上下文（如适用）

**审查模式：**

```typescript
// 好的错误处理
try {
  const config = JSON.parse(content);
  return config;
} catch (err) {
  throw new Error(`Failed to parse app-runtime.json: ${err.message}`);
}

// 不好的错误处理
const config = JSON.parse(content); // 没有捕获
```

## 关键风险点

### 高风险区域（必须重点审查）

1. **编辑口令哈希存储**
   - 文件：`src/lib/editor-auth-runtime.ts`
   - 风险：明文存储口令会导致严重安全问题
   - 验证：确认只存储哈希值

2. **数据目录热切换**
   - 文件：`src/lib/runtime-config.ts`
   - 风险：热切换可能导致数据分裂或丢失
   - 验证：确认进入 pending 状态而非直接切换

3. **配置文件原子写入**
   - 文件：配置保存相关逻辑
   - 风险：非原子写入可能导致配置损坏
   - 验证：使用临时文件 + rename 模式

4. **初始化竞态条件**
   - 文件：`src/app/api/setup/route.ts`
   - 风险：并发初始化可能导致重复或不一致
   - 验证：确认有锁或检查机制

5. **环境变量优先级**
   - 文件：所有配置读取逻辑
   - 风险：优先级错误可能导致配置被意外覆盖
   - 验证：文件 > 环境变量 > 默认值
