# 分阶段优化路线图

生成日期：2026-06-07
目标：把审查结论转成可交给后续 coding agent 的分阶段任务。默认不改业务方向：继续 Next.js、自带后台、本地文件持久化、R2 明文备份、Docker/GHCR 发布。

## 阶段划分原则

- 第 0 阶段：防止硬约束回归，立即修文档和测试护栏。
- 第 1 阶段：发布前必须修复的工程风险。
- 第 2 阶段：后台 UX 和可访问性。
- 第 3 阶段：数据架构一致性。
- 第 4 阶段：长期演进，按数据规模和使用场景触发。

## 第 0 阶段：立即修复

### 0.1 固化 R2 明文备份规则

目标：任何后续修改都不能把 R2 远端备份改回口令式方案。

收益：避免再次出现用户无法直接检查/恢复 R2 JSON 的回归。

风险：无业务风险。

涉及文件：

- `AGENTS.md`
- `README.md`
- `tests/lib/r2-backup-storage.test.ts`
- `tests/app/setup-route.test.ts`
- `tests/app/cloudflare-r2-route.test.ts`

验证方式：

- 检查 docs、src、README、deploy 中没有已移除的远端备份口令设计相关文案。
- `npm run test:architecture`
- `npm run test -- tests/lib/r2-backup-storage.test.ts tests/app/setup-route.test.ts tests/app/cloudflare-r2-route.test.ts`

预计工作量：0.5 天。

### 0.2 固化 Docker 更新保留 `.env data`

目标：更新镜像不覆盖生产配置和数据。

收益：用户更新后不需要重新配置，不丢数据。

风险：脚本测试需要 fake docker/git/tar，注意不要真实操作本机 Docker。

涉及文件：

- `README.md`
- `compose.yaml`
- `deploy/compose.prod.yaml`
- `deploy/git-deploy.sh`
- `.github/workflows/docker-deploy.yml`
- `tests/architecture/repository-structure.test.ts`

验证方式：

- 静态断言 bind mount。
- 新增 `tests/scripts/docker-update.test.ts` 行为级测试。
- `npm run test:architecture`。

预计工作量：1 天。

### 0.3 文档索引更新

目标：让后续 agent 能直接从中文文档进入任务。

收益：减少重复审查。

风险：无。

涉及文件：

- `docs/优化文档索引.md`
- 本次 8 份优化文档。

验证方式：

- `git diff --check`
- docs 链接人工快速检查。

预计工作量：0.5 天。

## 第 1 阶段：发布前修复

### 1.1 Docker 同源镜像 smoke

目标：测试镜像和发布镜像使用同一 Buildx 构建产物/缓存。

收益：消除“测的不是发的”。

风险：workflow 改动较大，需要一次只改 Docker job。

涉及文件：

- `.github/workflows/docker-deploy.yml`
- `.github/workflows/ui-smoke.yml`
- `scripts/test/*.py`

验证方式：

- GitHub Actions 运行成功。
- Docker smoke 日志显示挂载临时数据目录。
- GHCR digest 输出。

预计工作量：1.5 天。

### 1.2 Docker smoke 挂载真实数据目录

目标：CI 中验证容器对 `/var/lib/blog-navigation` 的读写和重启保留。

收益：提前发现权限、entrypoint、volume 问题。

风险：需要清理临时容器和目录。

涉及文件：

- `.github/workflows/docker-deploy.yml`
- `Dockerfile`
- `deploy/docker-entrypoint.sh`

验证方式：

- CI docker run 包含 `-v "$PWD/.tmp/docker-smoke-data:/var/lib/blog-navigation"`。
- 容器重启后 marker 仍存在。

预计工作量：1 天。

### 1.3 digest 部署与 `.last-good-digest`

目标：生产部署和回滚使用不可变 digest。

收益：回滚目标确定，避免 `latest` 漂移。

风险：需要兼容首次部署没有 last-good 的场景。

涉及文件：

- `.github/workflows/docker-deploy.yml`
- `deploy/git-deploy.sh`
- `deploy/compose.prod.yaml`
- `README.md`

验证方式：

- deploy summary 显示 new digest 和 previous digest。
- 健康检查失败时回滚到 previous digest。
- `.env data` 不变。

预计工作量：1.5 天。

### 1.4 Trivy 镜像扫描

目标：HIGH/CRITICAL 镜像漏洞阻断发布。

收益：供应链风险可见。

风险：首次启用可能暴露基础镜像漏洞，需要允许临时 allowlist。

涉及文件：

- `.github/workflows/docker-deploy.yml`
- `.github/dependabot.yml` 或 Renovate 配置。

验证方式：

- Actions 输出 Trivy summary。
- SARIF 上传或 artifact 存档。

预计工作量：0.5 天。

### 1.5 轻量 health endpoint

目标：Dockerfile、compose、deploy 统一探 `/api/health`。

收益：首页 SSR 或内容问题不会误判进程健康。

风险：端点设计不能访问敏感数据。

涉及文件：

- `src/app/api/health/route.ts`
- `Dockerfile`
- `deploy/compose.prod.yaml`
- `.github/workflows/docker-deploy.yml`

验证方式：

- 单元测试 health 返回 200。
- Docker healthcheck 通过。

预计工作量：0.5 天。

预计工作量：2 天。

### 2.3 Loading 和 focus 语义

目标：loading 有 `role=status`，程序化定位有可见 focus。

收益：可访问性基础补齐。

风险：低。

- `src/app/editor/(authenticated)/blog/components/PreviewPane.tsx`

验证方式：

- `@testing-library/react` 查询 role。

### 2.5 R2 操作忙碌态

目标：同步、恢复、bootstrap 显示具体 busy action。

预计工作量：0.5 天。

## 第 3 阶段：架构优化

### 3.1 R2 pending failed 状态和手动重试

目标：失败任务不再静默丢弃。

收益：远端备份可靠性和可观测性提高。

风险：需要兼容旧 `.backup-pending.json`。

涉及文件：

- `src/lib/backup-coordinator.ts`
- `src/lib/editor-remote-backup.ts`
- `src/app/api/data/backup/remote/*`
- `src/app/editor/(authenticated)/settings/CloudflareR2SettingsPanel.tsx`
- `tests/lib/backup-coordinator.test.ts`

验证方式：

- 失败 3 次进入 failed。
- 手动重试后成功清除 failed。
- 设置页展示失败原因。

预计工作量：2 天。

### 3.2 snapshot 绑定 manifest/hash

目标：snapshot 语义准确。

收益：恢复和审计更可信。

风险：需要调整 backup payload version。

涉及文件：

- `src/lib/editor-data-backup.ts`
- `src/lib/editor-remote-backup.ts`
- `src/lib/r2-backup-storage.ts`
- `tests/lib/r2-backup-storage.test.ts`

验证方式：

- 保存时 snapshot key 含 manifest hash。
- 延迟 drain 时不伪装为旧状态。

预计工作量：1.5 天。

### 3.3 manifest transaction

目标：资源 JSON 与 manifest 事务一致。

收益：消除核心数据一致性风险。

风险：存储层改动较深，必须有失败注入测试。

涉及文件：

- `src/lib/editor-data-storage.ts`
- `src/lib/editor-data-lock.ts`
- 新增 `src/lib/manifest-transaction.ts`
- 新增 `src/lib/atomic-json-writer.ts`

验证方式：

- 资源写失败。
- manifest 写失败。
- rename 失败。
- 重启恢复 marker。

预计工作量：3 天。

### 3.4 配置写入 fsync 和锁

目标：R2 配置、运行时配置与主数据写入安全级别一致。

收益：配置损坏风险降低。

风险：路径兼容。

涉及文件：

- `src/lib/r2-backup-storage.ts`
- `src/lib/app-runtime-config.ts`
- `src/lib/runtime-config.ts`

验证方式：

- 配置写失败不留下半文件。
- 恢复期间不能并发改配置。

预计工作量：1 天。

### 3.5 schemaVersion 和迁移 registry

目标：为后续数据拆分和结构升级准备。

收益：备份恢复跨版本可控。

风险：旧数据兼容必须谨慎。

涉及文件：

- `src/lib/editor-data-storage.ts`
- `src/lib/editor-data-backup.ts`
- `data/manifest.json` 生成逻辑
- `tests/scripts/data-migration-scripts.test.ts`

验证方式：

- 旧 manifest 可读。
- 新 manifest 写 schemaVersion。
- backup payload 包含 schemaVersion。

预计工作量：2 天。

## 第 4 阶段：长期演进

### 4.1 文章分文件存储

目标：从单体 `articles.json` 演进到 `articles/index.json + articles/by-slug/*.json`。

收益：降低整包写入、hash、R2 payload 压力。

风险：迁移复杂，导入导出和恢复要兼容。

涉及文件：

- `src/lib/editor-data-storage.ts`
- `src/lib/markdown.ts`
- `src/lib/editor-data-backup.ts`
- `scripts/data/runtime-data.mjs`

验证方式：

- 旧单体数据迁移。
- 新分文件读写。
- R2 latest 仍是明文 JSON。
- 恢复后 hash 一致。

预计工作量：5 天。

### 4.2 轻量权限模型

目标：从单编辑口令演进到 admin/editor 两级能力。

收益：为多人协作预留路径。

风险：认证和 session 改动敏感。

涉及文件：

- `src/lib/editor-auth-runtime.ts`
- `src/lib/editor-api-auth.ts`
- `src/app/editor/*`

验证方式：

- admin 可改设置。
- editor 可改文章/导航。
- 未授权 API 403。

预计工作量：3 天。

### 4.3 schema 驱动字段

目标：文章、导航、设置字段定义集中。

收益：减少表单和数据模型漂移。

风险：过度抽象。

涉及文件：

- `src/lib/resource-schema/*`
- 编辑器表单组件。

验证方式：

- schema 单测。
- 表单默认值和错误来自 schema。

预计工作量：4 天。

### 4.4 R2 snapshot 保留策略

目标：长期启用 snapshot 不无限增长。

收益：控制 R2 成本和对象数量。

风险：删除策略必须谨慎。

涉及文件：

- `src/lib/r2-backup-storage.ts`
- `src/app/api/data/backup/remote/*`

验证方式：

- list snapshot。
- dry-run 删除。
- 保留最近 N 份和最近 N 天。

预计工作量：2 天。

## 推荐实现顺序

1. Docker 更新行为级测试。
2. Docker 同源镜像 smoke + volume smoke。
3. digest 部署 + `.last-good-digest`。
4. Trivy。
5. UX P1：对比度、runtime 表单错误、loading。
6. R2 failed 队列。
7. snapshot 绑定 manifest/hash。
8. manifest transaction。
9. schemaVersion。
10. 文章分文件存储。

## 每次实施后的共同验证

```bash
npm run lint -- --fix
npm run typecheck
npm run test:coverage
npm run test:architecture
git diff --check
```

涉及 Docker/GHCR 的变更优先推 GitHub Actions 验证；本地不要构建发布镜像。
