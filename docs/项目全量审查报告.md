## 一、总体结论

blog-navigation 是一个**工程成熟度显著高于同类个人项目**的 Next.js 博客 + 导航站 + 后台编辑器。它在「不引入数据库」的前提下，自研了一套带 manifest 乐观锁、原子写、文件锁、崩溃恢复状态机的文件型持久化层，并配套了 CSP nonce、CSRF 双 token、登录限流、rehype-sanitize 等纵深安全防御，以及 pin-to-SHA、provenance/SBOM、digest 部署、失败自动回滚的发布链路。43 个测试文件 + 架构约束测试 + UI smoke 构成了远超平均水平的质量基线。

但审查也暴露出**几类结构性风险**，集中在三个方向：

1. **数据安全的「最后一公里」不可靠**：默认零异地备份、R2 备份明文且 fire-and-forget 易丢、快照只增不减、恢复事务的回滚源未 fsync。一旦数据卷损坏，默认配置下等于全量丢失。
2. **发布链路存在「测的不是发的、配置与镜像不同源」的断层**：CI 双重构建、compose.prod.yaml 从不被投递到生产机、回滚依赖运行时推断、缺镜像层 CVE 扫描。
3. **测试存在「always-pass 弱断言」与关键路径空白**：CSRF/限流测试同时接受相反结果等于没测，Setup 初始化与客户端数据 hooks 覆盖率极低或为零。

此外有一个**文档治理问题**：三份时间点快照型审计文档已经归档到 `docs/archive/`，后续应以本报告和 ADR 为准，避免历史快照与当前代码状态混用。

**一句话定级**：架构与安全基线优秀，但「数据容灾默认不设防」「发布链路有断层」「部分测试是假绿」三项必须在下一次正式发布前处理。

## 三、当前项目架构说明

### 3.1 分层架构

```
公开访问层  /  /blog  /navigation  /posts/[slug]      （SSR/RSC，异步磁盘读 + 2s 缓存）
   │
中间件层   middleware.ts：CSP nonce + strict-dynamic、HSTS、/editor session 校验
   │
编辑器层   /editor /editor/login /editor/blog /editor/navigation /editor/settings
   │
API 层     /api/editor-auth  /api/data/{articles,navigation,settings,backup}  /api/setup  /api/runtime-config  /api/search
   │        鉴权：ensureEditorWriteRequest = session + same-origin + CSRF；并发：writeXxxToDiskIfRevisionMatches 乐观锁
   │
存储核心   editor-data-storage.ts：文件锁(editor-data-lock.ts) + 原子写(temp→fsync→rename→dir fsync) + manifest(sha256+revision) + 恢复状态机(.restore-state.json)
   │
本地磁盘   BLOG_DATA_ROOT：articles/articles.json  navigation/tools.json  settings/site.json  manifest.json  settings/{cloudflare-r2,editor-auth}.json
   │        写后异步 queueCurrentBackupToRemote（fire-and-forget）
   │
远端 R2    latest/backup.json（明文）  snapshots/YYYY/MM/DD/<ts>-<reason>.json
```

种子数据（随镜像只读）：`content/seeds/posts/*.md`、`content/seeds/navigation/data/tools.json`。读取时运行时数据源（JSON）优先，fallback 到种子（Markdown）。

### 3.3 主要数据流（5 条）

1. **文章发布流**：编辑器 PUT `/api/data/articles`（带 revision）→ CSRF 校验 → `parseArticlesData` → 文件锁内比对 revision（冲突 409）→ 原子写 + fsync → 更新 manifest → 异步触发 R2 备份 → 返回新 revision。证据：`src/app/api/data/articles/route.ts:58-137`、`editor-data-storage.ts:869-899`。
2. **公开文章读取流**：`/posts/xxx` → `getPostBySlugArrayAsync` → 检查运行时数据源 → 运行时读 `articles.json` / 种子读 Markdown → 过滤 `published` → `MarkdownContent` 渲染。证据：`src/lib/markdown.ts:506-521`。
3. **R2 备份流**：写成功 → `queueCurrentBackupToRemote` 入队（单飞合并）→ `createCurrentEditorBackupPayload` → PUT snapshot? + PUT latest。证据：`editor-remote-backup.ts:75-122`、`r2-backup-storage.ts:486-538`。
4. **首次设置流**：`/setup`（`isApplicationSetupComplete()` false）→ PUT `/api/setup` → 解析配置 → bootstrap R2 → `initializeRuntimeEditorAuth` 写 `editor-auth.json` → `saveAppRuntimeConfig(markSetupComplete)` → 自动登录。证据：`src/app/api/setup/route.ts:195-341`。
5. **配置优先级链**：`app-runtime.json` > 环境变量 > 默认值（publicSiteUrl/cookieSecure/trustedProxyIps）；R2：`cloudflare-r2.json` > `R2_*` env > disabled。证据：`app-runtime-config.ts:181-225`、`r2-backup-storage.ts:224-255`。

## 五、安全问题

**已做到的纵深防御（值得肯定，避免后续误删）**：
- `middleware.ts:19-35` CSP nonce + strict-dynamic，生产 HSTS；
- `next.config` X-Frame-Options/nosniff/Referrer-Policy/Permissions-Policy；
- 编辑后台 session cookie + 登录限流 + CSRF 双 token + Origin 校验 + timing-safe 比较；
- 公开渲染 rehype-sanitize + skipHtml 防 XSS；open-redirect 防护；
- 生产 compose `read_only` + `tmpfs` + `cap_drop ALL` + `no-new-privileges`；非 root（uid 1001）；
- 所有第三方 Action pin 到 40 位 commit SHA；provenance + sbom。

> ✅ **纠正既有审计漂移**：历史审计快照已迁入 `docs/archive/`，安全项状态应以当前代码和测试为准。

## 六、数据安全问题（详见专项方案）

核心风险已在 P0/P1 列出。补充并发与损坏维度：

## 八、CI/CD 问题（详见专项方案）

- **部署链路断层**（P1-1）：compose.prod.yaml 从不投递生产机；部署仅 `workflow_dispatch && deploy==true` 触发，存在「以为推 tag 就上线」认知偏差；两条部署链路（workflow SSH vs `git-deploy.sh`）行为不同且并存。
- **构建浪费**（P1-3）：verify 的 `docker build --tag smoke` 无 cache-from/to 纯冷构建，build-and-push 再构建一次。
- **smoke 不可信**：CI smoke 用裸 `docker run`（无 read_only/cap_drop），与生产 compose 不等价；仅 curl `/blog /navigation` 不校验内容，200 错误页也算过；健康判定不查容器 HEALTHCHECK 状态。
- **回滚脆弱**（P1-2）：依赖 `docker inspect` 推断 PREV_DIGEST，`git-deploy.sh` 回滚目标可能是可变 tag 等于没回滚。
