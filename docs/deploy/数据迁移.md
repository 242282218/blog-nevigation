# 数据迁移

运行时数据按可迁移设计。旧服务器可访问时，优先直接复制 `data/`；无法直接复制时，再使用 JSON 备份包、GitHub 加密备份或 R2 云恢复。

## 适用场景

适用于更换服务器、从备份恢复、从本地迁移到生产环境，或在不同部署目录之间搬迁运行时数据。

迁移目标是恢复以下数据：

- 文章数据。
- 导航数据。
- 站点设置。
- 编辑器运行时认证设置。
- `manifest.json`。

R2 凭据和 GitHub 备份密钥不应放进普通数据备份包。密钥应通过 `.env`、服务器密钥管理或部署平台 secrets 单独管理。

## 推荐方案：服务器到服务器迁移

旧服务器可访问时，优先使用该方案。它保留 `data/`、`.env` 和 `compose.prod.yaml`，最接近原环境。

在旧服务器上执行：

```bash
cd /opt/blog-nevigation
docker compose -f compose.prod.yaml stop app
tar -czf blog-navigation-data.tgz data .env compose.prod.yaml
```

把压缩包复制到新服务器后执行：

```bash
mkdir -p /opt/blog-nevigation && cd /opt/blog-nevigation
tar -xzf /path/to/blog-navigation-data.tgz
docker compose -f compose.prod.yaml pull
docker compose -f compose.prod.yaml up -d
```

启动后验证：

```bash
# 从源码 checkout 中执行，不要在只有部署文件的生产目录中执行。
npm run data:verify -- /opt/blog-nevigation/data

cd /opt/blog-nevigation
HEALTHCHECK_PORT=$(docker compose -f compose.prod.yaml port app 3000 | awk -F: 'END {print $NF}')
curl -I "http://127.0.0.1:${HEALTHCHECK_PORT:-3000}/"
```

成功标准：

- `npm run data:verify` 通过。
- 首页和主要公开页面可访问。
- `/editor` 可登录，文章、导航和站点设置完整。
- 如果启用了 R2，`/editor/settings` 中的 R2 状态正常。

## 备份包迁移

适用于无法直接复制服务器目录、只需要迁移业务数据的场景。

导出：

```bash
npm run data:export -- ./data ./output/blog-navigation-backup.json
```

导入：

```bash
npm run data:import -- ./output/blog-navigation-backup.json ./data
npm run data:verify -- ./data
```

备份包包含文章、导航、站点设置和 manifest。旧版备份包如果没有站点设置，会恢复默认设置。

备份包不包含 `data/settings/cloudflare-r2.json` 或 R2 凭据。除非迁移完整的 `data/` 目录和 `.env`，否则新服务器需要重新配置 R2。

导入和 `data:verify` 会执行与应用一致的公开导航约束：分类 slug 必须唯一，生产工具 URL 必须是 HTTPS，每个工具至少要有一个 tag。无效备份包会在替换目标运行时数据前失败。

## GitHub 加密备份迁移

适用于备份文件存放在私有 GitHub 仓库或 GitHub Actions artifact 中的场景。

恢复时必须使用创建备份时的原始密钥：

```bash
export GITHUB_BACKUP_ENCRYPTION_KEY='replace-with-the-original-secret'
npm run data:restore:encrypted -- ./backups/blog-navigation-backup.enc.json ./data
npm run data:verify -- ./data
```

创建加密备份：

```bash
export GITHUB_BACKUP_ENCRYPTION_KEY='replace-with-a-long-random-secret'
npm run data:backup:github -- ./data
```

如果 `GITHUB_BACKUP_ENCRYPTION_KEY` 丢失，已加密的备份无法恢复。不要把该密钥提交到源码仓库或备份仓库。

## 修复 manifest

如果复制过来的数据目录早于 `manifest.json`，可以重新生成：

```bash
npm run data:verify -- ./data --write-manifest
npm run data:verify -- ./data
```

只有在确认源数据完整时才使用 `--write-manifest`。如果文章或导航数据本身已经缺失，重写 manifest 不能恢复丢失数据。

## R2 云恢复迁移

本地数据不可用时，可以从 R2 恢复：

1. 部署一台新的服务器，并确认 `BLOG_DATA_ROOT` 指向持久化目录。
2. 在 `.env` 中配置与备份时一致的 R2 bucket、prefix 和 `R2_BACKUP_ENCRYPTION_KEY`。
3. 启动容器。
4. 登录 `/editor`。
5. 执行云恢复。
6. 执行公开页面 smoke check。
7. 从源码 checkout 中执行 `npm run data:verify -- /opt/blog-nevigation/data`。

如果 `BLOG_DATA_ROOT/settings/cloudflare-r2.json` 已存在，它会完整优先于 `.env`。恢复失败且 `.env` 看起来正确时，优先检查该设置文件是否损坏、缺字段或指向了错误 bucket。

## 迁移前检查清单

- 已确认旧服务器数据目录位置。
- 已确认是否启用 R2，以及是否保存了 `R2_BACKUP_ENCRYPTION_KEY`。
- 已确认是否使用 GitHub 加密备份，以及是否保存了 `GITHUB_BACKUP_ENCRYPTION_KEY`。
- 已确认新服务器部署目录为 `/opt/blog-nevigation` 或已同步修改所有命令中的路径。
- 已准备回滚方案：保留旧服务器数据或保留迁移前压缩包。

## 迁移后检查清单

- 数据校验通过。
- 首页和主要文章页面可访问。
- 编辑器可登录。
- 新增或编辑一条测试数据后可以正常保存。
- 如果启用 R2，手动同步和恢复状态正常。
