# 服务器部署

本文是项目的生产部署主路径。推荐做法是：代码推送到 `main` 后，由 GitHub Actions 构建并发布 GHCR 镜像；服务器只拉取部署文件和镜像，再通过 Docker Compose 启动应用。

## 适用场景

适用于公网或内网生产服务器部署。服务器只需要 Docker、Docker Compose、Git 和基础 Shell 环境，不需要安装 Node.js 依赖，也不需要在服务器上执行 `npm install`、`npm run build` 或 `docker build`。

不适用于本地开发调试。本地 Docker 用法见 [Docker 补充说明](Docker补充说明.md)。

## 部署架构

```text
GitHub main -> GitHub Actions -> GHCR image -> server Docker Compose
                                      |
                                      v
                         git-deploy.sh 更新部署文件
                                      |
                                      v
                         ./data 挂载为 /var/lib/blog-navigation
```

Cloudflare 可以放在服务器前面提供 DNS、TLS 和 CDN。运行时数据仍以服务器文件系统为主，必要时再同步到 Cloudflare R2 做灾备。

服务器不构建应用镜像。服务器只保留一个受控的 Git checkout 用于读取部署文件，并拉取 GitHub Actions 已经构建好的镜像。

## 首次部署

在服务器上执行：

```bash
mkdir -p /opt/blog-nevigation && cd /opt/blog-nevigation

curl -fsSL https://raw.githubusercontent.com/242282218/blog-nevigation/main/deploy/git-deploy.sh \
  -o /opt/blog-nevigation/git-deploy.sh
chmod +x /opt/blog-nevigation/git-deploy.sh

EDITOR_ACCESS_TOKEN="$(openssl rand -base64 32)"

cat > .env <<EOF
EDITOR_ACCESS_TOKEN=${EDITOR_ACCESS_TOKEN}
APP_PORT=3000
NEXT_PUBLIC_SITE_URL=https://your-domain.example
COOKIE_SECURE=true
TRUSTED_PROXY_IPS=
R2_BACKUP_ENABLED=false
EOF

DEPLOY_PATH=/opt/blog-nevigation /opt/blog-nevigation/git-deploy.sh
```

如果首次启动发生在只支持 HTTP 的私有主机上，可以临时设置 `COOKIE_SECURE=false`。在通过 HTTPS 暴露编辑器之前，必须改回 `COOKIE_SECURE=true`。

`NEXT_PUBLIC_SITE_URL` 必须填写生产公网访问地址。它用于生成 metadata、robots 和 sitemap；不设置时会退回 localhost，只适合本地调试。

`TRUSTED_PROXY_IPS` 应填写应用前一层反向代理的直接来源 IP。登录和搜索限流只信任来自这些代理的转发客户端 IP。只有应用直接接收容器流量时才留空。

`EDITOR_ACCESS_TOKEN` 是最简单的生产认证方式。如果想从登录页初始化编辑器密码，可以不设置 `EDITOR_ACCESS_TOKEN`，改为设置一次性的 `EDITOR_RUNTIME_AUTH_SETUP_TOKEN`，启动容器后打开 `/editor/login`，输入 setup token 和新的编辑器密码。不要在公开生产环境中启用 `EDITOR_ALLOW_RUNTIME_AUTH_SETUP=true`，除非该主机已经通过其它方式隔离。

使用 `EDITOR_ACCESS_TOKEN` 登录时，随机会话状态会写入 `BLOG_DATA_ROOT`。如果运行多个应用副本，所有副本必须挂载同一个数据目录；否则应使用粘性路由，或只运行单副本。

## 更新部署

常规更新仍然执行同一个部署脚本：

```bash
DEPLOY_PATH=/opt/blog-nevigation /opt/blog-nevigation/git-deploy.sh
```

默认情况下，脚本会部署当前 Git commit 对应的不可变镜像 tag，例如 `main-<7-char-sha>`。只有明确想使用分支浮动 tag 时，才设置 `IMAGE_TAG=main`：

```bash
IMAGE_TAG=main DEPLOY_PATH=/opt/blog-nevigation /opt/blog-nevigation/git-deploy.sh
```

## 健康检查

部署后在服务器上检查容器状态、日志和 HTTP 响应：

```bash
cd /opt/blog-nevigation
docker compose -f compose.prod.yaml ps
docker compose -f compose.prod.yaml logs --tail=100 app
HEALTHCHECK_PORT=$(docker compose -f compose.prod.yaml port app 3000 | awk -F: 'END {print $NF}')
curl -I "http://127.0.0.1:${HEALTHCHECK_PORT:-3000}/"
```

成功标准：

- `docker compose ps` 中 `app` 处于运行状态。
- 日志中没有持续重启、鉴权配置错误或数据校验错误。
- `curl -I` 返回 `200`、`301`、`302` 或其它符合预期的 HTTP 响应。
- 公开页面可访问，`/editor` 可正常登录。

## 运行时数据

生产部署目录应包含：

```text
compose.prod.yaml
.env
data/
repo/
```

`data/` 是迁移边界，包含文章数据、导航数据、站点设置、编辑器运行时认证设置和 `manifest.json`。

从源码 checkout 中执行数据校验，并把生产数据目录作为参数传入：

```bash
npm run data:verify -- /opt/blog-nevigation/data
```

注意：仓库目录名使用 `blog-nevigation`，应用镜像、数据文件和备份文件中可能使用 `blog-navigation`。文档中的路径按当前仓库和部署脚本保持不变，不要自行混改目录名。

## GitHub Actions 自动部署

配置以下 GitHub Actions secrets 后，工作流可以通过 SSH 部署到服务器：

```text
DEPLOY_HOST
DEPLOY_USER
DEPLOY_SSH_KEY
DEPLOY_PATH
```

手动部署使用 `workflow_dispatch`，并设置 `deploy=true`。

也可以直接通过 SSH 在服务器侧触发部署：

```bash
ssh root@your-server 'DEPLOY_PATH=/opt/blog-nevigation /opt/blog-nevigation/git-deploy.sh'
```

部署任务会在构建没有发布不可变镜像 digest 时拒绝启动。Docker Compose 重启应用后，工作流会通过 `docker compose port app 3000` 获取实际映射端口并执行健康检查。如果新容器健康检查失败，只有在旧容器存在有效 `@sha256:` 镜像 digest 时才会回滚，并对回滚后的容器执行同样的健康检查。

## 常见问题排查

### 登录后立即失效

检查多个副本是否共享同一个 `BLOG_DATA_ROOT`。如果没有共享目录，改为单副本或配置粘性路由。

### 反向代理后限流 IP 不正确

检查 `TRUSTED_PROXY_IPS` 是否填写了反向代理直接连接应用的 IP，而不是用户公网 IP。

### R2 配置不生效

如果 `BLOG_DATA_ROOT/settings/cloudflare-r2.json` 存在，它会完整优先于 `.env` 中的 R2 变量。需要切回 `.env` 时，应通过 `/editor/settings` 删除或修改该配置文件。详见 [Cloudflare R2 备份](Cloudflare-R2备份.md)。
