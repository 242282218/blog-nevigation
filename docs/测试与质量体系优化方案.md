## 三、测试质量问题（假绿 & 脆弱 & 慢）

### 3.1 假绿弱断言（P1-7，最优先修）
> 这类测试**通过 ≠ 功能正确**，比没测更危险（给人虚假安全感）。

| 文件 | 问题 | 修复 |
|------|------|------|
| `tests/app/csrf-protection.test.ts` | 几乎所有用例 `expect([403,503]).toContain(status)` 同时接受「CSRF 失败」与「未配代理」；'accept valid CSRF' 用 `if(response){}else{}` 让两种相反结果都过 = always-pass | 设 `EDITOR_ACCESS_TOKEN`/`TRUSTED_PROXY_IPS`/`SKIP_IP_VALIDATION` 构造确定环境，断言精确 403；正向用例 mock `isValidRuntimeEditorSession` 返回通过 |
| `tests/app/rate-limit.test.ts` | `if(response){expect([429,503,null])}` 接受三种结果；'15-minute window' 未推进时间（无 fake timers）只断言首次 null，未验证窗口过期 | `vi.useFakeTimers` + `advanceTimersByTime` 真正验证 15min 过期；固定代理环境使状态码确定 |
| `tests/app/markdown-xss.test.tsx` | 'block dangerous protocol' 用 `if(link){}`：sanitize 移除链接时 link=null 直接通过，未验证任何东西 | 分两类断言：链接被移除（querySelector('a')===null）**或** href 被清洗为安全值，二者择一显式断言 |

### 3.2 脆弱测试
| 文件 | 问题 | 修复 |
|------|------|------|
| `tests/architecture/repository-structure.test.ts` | 单 it 含 100+ 硬编码字符串断言（Dockerfile/compose/workflow/README 逐行 toContain），文案微调即破坏，违反单一职责 | 拆分按关注点多用例；保留「禁止出现 X」安全约束，移除逐字文案断言，改结构化解析（解析 package.json/yaml 字段） |

### 3.3 flaky 风险
| 文件 | 问题 | 修复 |
|------|------|------|
| 多路由测试（editor-data-routes/remote-backup-route） | `process.chdir` + 改 `process.env` 隔离，抛错未走 afterEach 则 cwd/env 泄漏污染后续用例（顺序相关不稳定） | 优先依赖注入替代 chdir；env/cwd 恢复放 try/finally 或用 `vi.stubEnv` |

### 3.4 测试慢（P2-7）
- **问题**：全量 jsdom，44 文件 setup 累计 41.68s，纯 lib/scripts 也跑 jsdom，总耗时 ~43.5s。
- **修复**：`tests/lib` 和 `tests/scripts` 用 `// @vitest-environment node` 或 `environmentMatchGlobs` 配置，仅组件测试用 jsdom。
- **预期**：setup 时间显著下降。

## 五、mock 风险与治理（mock_risks）

| mock 目标 | 文件 | 风险 | 缓解 |
|-----------|------|------|------|
| `S3Client.send` | r2-backup-storage.test.ts | GetObject Body 用 string 而非 ReadableStream（sdkStreamMixin），掩盖流处理 bug；endpoint/签名绕过 | 补 MinIO 集成；mock Body 用 sdkStreamMixin 包装流 |
| `@/lib/r2-backup-storage` 整体 | remote-backup-route/editor-remote-backup | route→queue→storage 无端到端串联，错误码契约重新声明，签名漂移不被发现 | 增加不 mock 存储层、仅 mock S3Client 的集成用例 |
| `@/lib/editor-remote-backup` | editor-data-routes/remote-backup-route | 写 API 仅断言 mock 被以 {reason} 调用，未验真实入队副作用 | 保留契约验证 + 补集成验证真实队列状态 |
| `@/lib/editor-data-backup` | editor-remote-backup.test.ts | payload 构造被 mock，序列化/加密字段缺失不可见 | 至少一用例用真实 createCurrentEditorBackupPayload + 临时数据根 |
| `request.cookies`（Object.defineProperty） | csrf-protection.test.ts | 手工覆盖绕过真实 cookie 解析，大小写/编码分歧 | 用 createAuthedEditorRequest 或真实 Cookie header 让 NextRequest 自行解析 |

**治理原则**：保留单测的契约验证（快），另补**少量集成测试**（仅 mock 最外层 S3Client，串联真实 route→queue→storage）覆盖 mock 无法暴露的行为差异。

## 六、CI 阶段设计

### 6.1 当前问题
- verify 单 job 串行跑 audit/env/lint/typecheck/coverage/docker-smoke，反馈慢。
- 覆盖率阈值（60/50/57/61）低于实际值（67/59/67/67），无防回归护栏。
- 无 diff/增量覆盖率门禁，新增代码不被有效约束。

### 6.2 目标 CI 阶段（拆分并行）
```
PR / push 触发
├─ Job: lint+typecheck（并行）
├─ Job: unit test + coverage（并行，environmentMatchGlobs 加速）
│    └─ diff-coverage 门禁：新增代码行覆盖 ≥ 80%
├─ Job: architecture test（并行）
├─ Job: build + docker smoke（用 compose.prod.yaml，内容断言）
└─ Job: security（CodeQL + Trivy + gitleaks，见 Docker发布方案）
合并后 / 定时：
└─ Job: 集成测试（MinIO，测试机或 service container）
```

### 6.3 覆盖率门禁策略
- **立即**：阈值贴近当前实际锁基线（statements 67/branches 59/functions 67/lines 67），防倒退。
- **PR 级**：引入 diff coverage（如 `diff-cover`），新增/修改代码行覆盖 ≥ 80%，避免全局阈值被存量稀释。
