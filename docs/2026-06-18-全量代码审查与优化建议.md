## 项目逻辑梳理

### 运行模式

- Next.js 15 App Router + React 18 + TypeScript strict。
- 包管理器以 `package.json` 的 `packageManager: npm@11.6.2` 为准，当前仓库脚本全部走 npm。
- 数据默认存放在运行时数据根目录：本地为 `data/`，Docker 为 `/var/lib/blog-navigation`。
- 公开页面优先读取运行时数据；运行时文章文件不存在时，文章回退到 `content/seeds/posts`，导航回退到 `content/seeds/navigation/data/tools.json`。

### 核心数据流

- 编辑器前端通过 `useSyncedResource` 做本地 `localStorage` + 远端 API 同步。
- 文章、导航、站点设置分别走 `/api/data/articles`、`/api/data/navigation`、`/api/data/settings`。
- 三类核心数据写入都要求 revision 匹配，冲突时返回 409 并带回服务器版本。
- 服务端数据层通过 `.data-write.lock` 做跨进程互斥，通过临时文件 + rename + fsync 做原子写。
- 每次资源写入后更新 `manifest.json`，并刷新派生搜索索引。

### 备份与恢复

- R2 自动备份保持明文 JSON，符合项目级规则。
- R2 配置文件 `data/settings/cloudflare-r2.json` 优先于 `.env` 中的 R2 变量。
- 写入文章、导航、设置、媒体后会排队同步 R2；队列状态保存在 `.backup-pending.json`。
- 远端恢复前会先尝试写入 pre-restore snapshot，恢复时先恢复 JSON 数据和媒体清单，再按清单从 R2 拉取缺失媒体。

### 安全边界

- 编辑器 API 通过 session cookie 校验，写请求额外要求同源 Origin + CSRF 双提交 token。
- Markdown 渲染使用 `skipHtml` + `rehype-sanitize`，未发现 `dangerouslySetInnerHTML`。
- 生产环境首次初始化：没有 `EDITOR_ACCESS_TOKEN` 且没有 `EDITOR_RUNTIME_AUTH_SETUP_TOKEN` 时，初始化会被禁用。
- 登录和搜索限流基于进程内 Map，并依赖可信代理配置获得稳定客户端 IP。

## 高优先级问题

### P1-1 依赖安全审计失败，会阻断 CI verify

现象：`npm run audit:high` 当前失败，报告包括：

- `vite` 高危：Windows dev server / launch-editor 相关漏洞。
- `@babel/core` 低危：sourceMappingURL 任意文件读。
- `esbuild` 低危：Windows dev server 任意文件读。
- `js-yaml` 中危：merge key 重复 alias 导致 DoS，路径来自 `gray-matter` 相关依赖。

影响：

- `.github/workflows/docker-deploy.yml` 的 `verify` job 会执行 `npm run audit:high`，因此当前发布流水线会失败。
- Vite、esbuild、Babel主要影响开发/测试工具链，但 CI 门禁失败是实际阻断。

建议：

- 优先执行 `npm audit fix` 并复跑 `npm run check && npm run test:coverage && npm run build && npm run audit:high`。
- 如果 `gray-matter/js-yaml` 需要破坏性修复，不要直接 `--force` 降级；先评估 `gray-matter` 可升级版本或用 overrides 固定安全版本。
- 将依赖安全修复单独成 PR，避免和业务改动混在一起。

### P1-2 媒体 manifest 并发写入可能丢条目

位置：

- `src/lib/editor-media-storage.ts:316` 读取媒体 manifest。
- `src/lib/editor-media-storage.ts:337` 写回媒体 manifest。
- `src/app/api/data/media/route.ts:106` 调用 `storeEditorMediaFile()` 时没有进入统一数据根锁。

问题：

`storeEditorMediaFile()` 先读 manifest，再写文件，再合并并写 manifest。两个不同图片同时上传时，两个请求可能基于同一个旧 manifest 合并，最后写入者覆盖先写入者，导致某个媒体文件已存在但 manifest 丢失资产记录。文件 hash 去重只能解决同图重复上传，不能解决不同图片并发上传。

影响：

- R2 JSON 备份里的媒体清单不完整。
- 远端恢复时无法拉回 manifest 中缺失的媒体文件。
- 媒体一致性检查会出现 orphan file，但用户文章里仍可能引用该文件，后续迁移风险增加。

建议：

- 方案 A：在 `/api/data/media` 中用 `withRuntimeDataRootLock()` 包裹 `storeEditorMediaFile()`。改动最小，但图片写入会和文章/导航/设置写入串行。
- 方案 B：为 `media/manifest.json` 增加专用媒体锁。并发粒度更细，但锁模型变复杂。
- 推荐方案 A。当前项目明确单实例、JSON 文件存储优先，简单一致性比上传并发吞吐更重要。
- 补一个并发上传单测：并行保存两张不同图片，断言 manifest 最终有两个 asset。

## 中优先级问题

### P2-1 `/api/setup` GET 在初始化后仍公开返回运行时元信息

位置：`src/app/api/setup/route.ts:174-185`

问题：

`GET /api/setup` 不需要登录，即使 setup 已完成，也会返回 `config`、`editable`、`r2Settings`、`r2Status`。其中不含 R2 secret/access key，但会暴露数据目录、公开 URL、可信代理配置、R2 bucket/prefix/endpoint 等运维信息。

建议：

- setup 未完成时返回完整初始化所需信息。
- setup 完成后只返回 `{ setupCompleted: true }`，或要求编辑器 session 后再返回详细信息。
- 补测试：setup 已完成且无 session 时，不返回 `editable`、`r2Settings`、内部路径。

### P2-2 `EDITOR_ALLOW_RUNTIME_AUTH_SETUP` 配置残留，容易误导部署

位置：

- `deploy/compose.prod.yaml:31`
- `tests/app/setup-route.test.ts`
- `tests/app/editor-auth.test.ts`

问题：

运行时代码只读取 `EDITOR_RUNTIME_AUTH_SETUP_TOKEN`，不读取 `EDITOR_ALLOW_RUNTIME_AUTH_SETUP`。测试中已有“即使设置该变量也会被忽略”的用例，但生产 compose 仍暴露该环境变量，部署者可能误以为它能开启初始化。

建议：

- 从 `deploy/compose.prod.yaml` 删除 `EDITOR_ALLOW_RUNTIME_AUTH_SETUP`。
- 测试保留“旧变量无效”的兼容验证可以接受，但命名应明确为 legacy ignored。
- README 中只保留 `EDITOR_RUNTIME_AUTH_SETUP_TOKEN` 和 `EDITOR_ACCESS_TOKEN` 两条初始化路径。

### P2-3 搜索索引缺失时，兜底路径没有限制导航工具数量

位置：`src/app/api/search/route.ts:83-120`

问题：

搜索索引读取失败时会从源数据兜底扫描。文章有 `FALLBACK_MAX_DOCUMENTS = 500` 限制，但导航工具 `navigation.flatMap(...)` 没有限制。如果导航数据增长到数千或上万条，单次搜索会产生明显 CPU/内存压力。

建议：

- 对 tool documents 增加独立上限，例如 `FALLBACK_MAX_TOOLS = 1000`。
- 搜索索引缺失时优先尝试后台重建索引，而不是长期依赖兜底路径。
- 补测试：索引缺失且工具数量超限时，结果构建被截断并记录 warning。

### P2-4 Docker entrypoint 每次递归 chown 数据目录，媒体多时启动成本会上升

位置：`deploy/docker-entrypoint.sh:8-9`

问题：

容器每次启动都会 `chown -R nextjs:nodejs "$DATA_ROOT"`。数据目录只有 JSON 时成本很低；启用剪切板图片后，`data/media/files` 会持续增长，递归 chown 会拖慢重启、部署回滚和健康检查恢复。

建议：

- 首次启动或权限异常时才修复 ownership。
- 可用 marker 文件记录已修复状态，或只对顶层目录执行 `chown`，写失败时给出明确错误。
- 如果保留递归修复，至少在 README 中说明媒体量大时启动可能变慢。

### P2-5 远端恢复对媒体缺失采用 warning，可能留下破图

位置：`src/app/api/data/backup/remote/actions.ts` 中远端恢复流程。

问题：

当前流程会先恢复 JSON 和媒体 manifest，再拉取媒体文件。媒体恢复失败时接口仍返回 `success: true` 并带 `warnings`。这保留了数据恢复可用性，但会让文章中引用的图片变成 404。

建议：

- 给恢复按钮增加模式选择：严格恢复和尽力恢复。
- 严格恢复先预检查所有媒体对象可下载，再切换 JSON 数据。
- 尽力恢复保持当前行为，但 UI 要展示失败媒体路径和后续重试入口。

## 低优先级优化

### P3-1 `beforeunload` keepalive 保存大 JSON 不可靠

`useSyncedResource` 在页面关闭前会用 `fetch(..., keepalive: true)` 尝试提交完整文章/导航 JSON。浏览器对 keepalive body 有大小限制，大数据集下可能失败。当前已有本地保存和 revision 冲突保护，所以不是数据丢失主路径；建议后续改成显式“正在同步/同步完成”状态，不依赖关闭页面兜底。

### P3-2 公开数据源切换规则需要更明确

只要运行时 `articles/articles.json` 存在，公开文章就从运行时数据读取，不再混合 seeds。这个行为合理，但迁移初期容易让用户误以为种子文章丢失。建议在 README 或后台设置页说明“运行时文章源一旦初始化，会替代 seeds 文章源”。

### P3-3 覆盖率达标但局部盲区明显

当前全局覆盖率达标，但局部文件偏低：

- `src/lib/editor-media-remote.ts` 覆盖率 0。
- `src/app/api/data/media/route.ts` statement 52.5%、branch 28.57%。
- `src/app/api/data/cloudflare-r2/bootstrap/route.ts` statement 40.62%。
- `src/lib/markdown.ts` statement 47.73%。

建议优先补与生产风险相关的测试：媒体远端恢复、媒体上传错误路径、Cloudflare bootstrap API 错误映射、运行时文章源和 seeds 源切换。

### P3-4 npm 环境配置有噪声

多条 npm 命令输出：

```text
npm warn Unknown env config "state-dir". This will stop working in the next major version of npm.
npm warn Unknown env config "store-dir". This will stop working in the next major version of npm.
```

这不是项目代码错误，但会污染 CI 日志。建议检查用户级或全局 `.npmrc`，移除 npm 不识别的 `state-dir` / `store-dir` 配置。
