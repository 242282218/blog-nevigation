# 数据安全与备份恢复优化方案

生成日期：2026-06-07
硬约束：R2 远端备份保持明文 JSON；Docker 更新保留宿主机 `.env` 和 `data/`；更新前必须本地快照。

## 当前数据链路

```mermaid
flowchart TD
  UI[编辑器保存] --> API[PUT /api/data/articles|navigation|settings]
  API --> Check[鉴权 + CSRF + revision/hash]
  Check --> Lock[.data-write.lock]
  Lock --> Write[写资源 JSON]
  Write --> Manifest[更新 manifest.json]
  Manifest --> Queue[写入 R2 pending 队列]
  Queue --> Drain[启动 drain / 保存后 drain / 3 小时定时]
  Drain --> Payload[生成 backup payload]
  Payload --> R2Latest[R2 latest/backup.json 明文 JSON]
  Payload --> R2Snapshot[R2 snapshots 明文 JSON]
  Restore[R2 或本地恢复] --> Pre[恢复前本地快照]
  Pre --> RestoreState[恢复状态机]
  RestoreState --> Write
```

## 风险分析

### P1：数据文件与 manifest 双写非事务

风险：业务 JSON 已写入、manifest 写失败时，客户端收到失败但数据已变化；恢复前置条件和 hash 可能失真。

证据：`src/lib/editor-data-storage.ts:893`、`src/lib/editor-data-storage.ts:897`、`src/lib/editor-data-storage.ts:1047`。

修复：

1. 引入 `manifest-transaction.ts`。
2. `stage/articles.json.next`、`stage/manifest.json.next` 同时写入。
3. fsync 文件和目录。
4. rename 资源文件和 manifest。
5. 写入失败保留 `transaction-state.json`。
6. 启动时检测 marker，完成 commit 或 rollback。

验证：

- 注入 manifest 写失败，断言不会出现资源新、manifest 旧的静默状态。
- 注入进程中断，重启后 marker 可诊断。

### P1：R2 pending 任务不是写入时快照

风险：用户以为 snapshot 对应某次保存，但 drain 延迟后上传的是后续状态。

证据：`src/lib/backup-coordinator.ts:10`、`src/lib/editor-remote-backup.ts:128`、`src/lib/editor-remote-backup.ts:187`。

修复：

- latest 任务：合并去重，始终上传当前状态。
- snapshot 任务：入队时记录 `manifestHash`、`resourceHashes`、`createdAt`、`reason`。
- drain 前重新读取 manifest，如果 hash 不匹配，则 snapshot key 标记为 `delayed-current` 或重新生成“当前状态 snapshot”，不能伪装为旧写入时刻。

### P1：pending 失败静默丢弃

风险：R2 远端备份永久落后，用户无感。

证据：`src/lib/backup-coordinator.ts:8`、`src/lib/backup-coordinator.ts:238`。

修复：

- 队列状态从 `pending[]` 扩展为 `pending[] + failed[]`。
- failed 记录 lastError、attempts、lastAttemptAt、nextRetryAt。
- 设置页显示队列状态。
- 提供手动重试。
- 失败不阻塞本地保存，但必须可见。

### P2：配置类文件弱于主数据

风险：R2 配置和运行时配置损坏后，备份/恢复不可用。

证据：`src/lib/r2-backup-storage.ts:143`、`src/lib/r2-backup-storage.ts:148`、`src/lib/app-runtime-config.ts:275`。

修复：

- 配置写入复用 `atomic-json-writer.ts`。
- 写入 settings 目录 fsync。
- 配置写入也走数据根锁，避免与恢复并发。

### P2：脚本导入恢复弱于应用内恢复

风险：命令行导入中断后，恢复能力弱于 Web API。

证据：`scripts/data/runtime-data.mjs:414`、`scripts/data/runtime-data.mjs:494`、`scripts/data/runtime-data.mjs:518`。

修复：

- 脚本复用应用内 backup payload 校验规则。
- 导入前创建 `pre-script-import` 快照。
- 写入 `restore-state.json`。
- 失败时打印明确恢复命令。

### P2：Docker 回滚只回镜像不回数据

风险：新镜像启动后写坏数据，自动回滚镜像仍挂载同一份坏数据。

证据：`README.md:218`、`README.md:232`、`deploy/git-deploy.sh:197`、`deploy/git-deploy.sh:211`、`.github/workflows/docker-deploy.yml:353`、`.github/workflows/docker-deploy.yml:369`。

策略：

- 自动回滚只回镜像，不自动覆盖数据。
- 更新前 `.tgz` 本地快照保留。
- 数据回滚必须人工执行显式命令，并提示会覆盖当前 `data/`。
- 文档区分“镜像回滚”和“数据恢复”。

## R2 明文备份策略

### 不变量

- R2 对象内容是可读 JSON。
- 上传 ContentType 为 JSON。
- 下载后直接 parse。
- 不新增口令字段。
- 不把 Docker 本地 `.tgz` 当作 R2 替代品。

### Payload 格式

推荐 payload：

```json
{
  "version": 2,
  "schemaVersion": 1,
  "createdAt": "2026-06-07T00:00:00.000Z",
  "source": "blog-navigation",
  "reason": "article-save",
  "manifest": {
    "revision": "rev",
    "hash": "hash",
    "resources": {
      "articles": "sha256",
      "navigation": "sha256",
      "settings": "sha256"
    }
  },
  "data": {
    "articles": [],
    "navigation": {},
    "settings": {}
  }
}
```

说明：

- `version` 是 backup payload 版本。
- `schemaVersion` 是业务数据 schema。
- `manifest.resources` 用于恢复后重算校验。
- `data` 必须保留明文结构。

### R2 key 设计

- latest：`<prefix>/latest/backup.json`
- snapshot：`<prefix>/snapshots/YYYY/MM/DD/HHmmss-<reason>-<manifestHash>.json`
- failed metadata 可只在本地队列，不写 R2。

### 大小阈值

当前 R2 单体备份会随文章量增长。建议：

- payload 超过 1MB：设置页提示。
- payload 超过 4MB：保存成功但提醒拆分文章存储。
- 接近 5MB：禁止开启每次保存 snapshot，只保留 latest 和定时 snapshot。

## 本地备份策略

### Docker 更新前快照

必须包含：

- `.env`
- `data/`

当前证据：`README.md:218`、`deploy/git-deploy.sh:115`、`.github/workflows/docker-deploy.yml:312`。

保留策略：

- 默认保留最近 10 份。
- 文件名含时间戳。
- 不包含镜像层。
- 不上传到 R2。

### 应用内恢复前快照

触发场景：

- 从 R2 恢复。
- 从本地 backup payload 恢复。
- 脚本导入运行时数据。
- 数据 schema 迁移。

快照命名：

- `data/backups/pre-remote-restore-<timestamp>.json`
- `data/backups/pre-local-restore-<timestamp>.json`
- `data/backups/pre-migration-v<from>-to-v<to>-<timestamp>.json`

## manifest 校验

推荐 manifest：

```json
{
  "schemaVersion": 1,
  "revision": "rev",
  "updatedAt": "2026-06-07T00:00:00.000Z",
  "resources": {
    "articles": {
      "path": "articles/articles.json",
      "hash": "sha256",
      "revision": "rev"
    },
    "navigation": {
      "path": "navigation/tools.json",
      "hash": "sha256",
      "revision": "rev"
    },
    "settings": {
      "path": "settings/site.json",
      "hash": "sha256",
      "revision": "rev"
    }
  }
}
```

校验点：

- 保存后重读资源并校验 hash。
- 恢复前校验 backup manifest。
- 恢复后校验本地 manifest。
- R2 latest 下载后先校验，不直接覆盖。
- schemaVersion 不匹配时必须走迁移。

## 损坏文件处理

### JSON 解析失败

处理：

1. 不覆盖损坏文件。
2. 写诊断到 `data/diagnostics/`.
3. 设置页显示损坏文件路径和恢复入口。
4. 提供从本地 snapshot 或 R2 latest 恢复。

### manifest 与资源 hash 不一致

处理：

1. 页面进入只读保护或后台警告。
2. 提供“重建 manifest”按钮，但必须先快照。
3. 重建前展示差异。

### pending 队列损坏

处理：

1. 备份损坏的 `.backup-pending.json`。
2. 初始化空队列。
3. 立即排队一次 latest。
4. 设置页提示用户 R2 状态可能滞后。

## 2026-06-08 多 Agent 数据安全增补

本节是数据安全与备份恢复的最新执行方案。硬约束不变：R2 自动备份保持明文 JSON，Docker 更新保留 `.env` 和 `data/`。

### 当前数据链路复核

| 链路 | 当前实现 | 证据 |
|---|---|---|
| 数据根 | `BLOG_DATA_ROOT` 或默认路径 | `src/lib/runtime-config.ts:18` |
| Docker 数据目录 | `/var/lib/blog-navigation` bind mount | `compose.yaml:10`、`compose.yaml:24`、`deploy/compose.prod.yaml:26`、`:45` |
| 原子 JSON | temp/write/fsync/rename 基础 | `src/lib/atomic-json-writer.ts:30` |
| 数据写锁 | 跨进程 lock | `src/lib/editor-data-lock.ts:199` |
| manifest 事务 | 有事务状态和恢复 | `src/lib/manifest-transaction.ts:91`、`:122` |
| R2 上传 | `JSON.stringify(payload)` 明文上传 | `src/lib/r2-backup-storage.ts:492`、`:519` |
| R2 下载 | 下载后直接 `JSON.parse` | `src/lib/r2-backup-storage.ts:500` |
| 远端恢复 | currentManifest + pre-remote snapshot | `src/app/api/data/backup/remote/actions.ts:171`、`:191`、`:201` |
| 本地导入 | currentManifest 前置，缺持久 pre-local snapshot | `src/app/api/data/backup/route.ts:96`、`:137`、`src/lib/editor-data-storage.ts:1227` |

### 风险分析

#### 风险 1：R2 未启用时写入仍成功

`queueCurrentBackupToRemote` 在未配置时返回 disabled，业务数据已本地提交。证据：`src/lib/editor-remote-backup.ts:98`、`src/app/api/data/articles/route.ts:128`。这符合“R2 可选”的产品策略，但必须在 UI 状态中标为高风险，而不是让用户误以为已远端灾备。

处理：

- 设置首页和 R2 面板展示 `远端灾备：未启用`。
- 写入 API 返回 `backup.status = disabled` 时，前端显示非阻断风险提示。
- 生产部署文档将 R2 未启用列为“只具备本地备份”。

#### 风险 2：latest 覆盖写导致坏数据扩散

生产默认 `R2_SNAPSHOT_ON_WRITE=false`，latest 会覆盖。证据：`deploy/compose.prod.yaml:44`、`src/lib/editor-remote-backup.ts:118`、`src/lib/r2-backup-storage.ts:541`。

处理：

- 推荐生产设置 `R2_SNAPSHOT_ON_WRITE=true`。
- latest 继续覆盖，用于快速恢复最近状态。
- snapshot 按时间和 manifest/hash 命名，用于灾备恢复点。
- README 和运维文档说明 R2 bucket 生命周期保留策略。

#### 风险 3：队列文件缺跨进程锁与损坏隔离

`enqueuePendingBackupTask` 是读旧状态再写新状态，未复用编辑数据锁。证据：`src/lib/backup-coordinator.ts:339`、`:343`、`src/lib/editor-data-lock.ts:199`。`readState` 解析失败抛错，不自动隔离。证据：`src/lib/backup-coordinator.ts:159`、`:166`。

处理：

- 新增 `withBackupQueueLock`，用 lockfile 包裹 read/modify/write。
- `readState` 解析失败时，把原文件移动为 `.backup-pending.corrupt-<timestamp>.json`。
- 重建空队列并返回 `corruptRecovered` 状态。
- 设置页展示“已隔离损坏队列，需要立即手动同步”。

#### 风险 4：恢复提交后目录 fsync 与校验不足

恢复事务中逐个 rename 后写 committed 状态，缺目标目录 fsync；committed 恢复分支只清理状态，不校验。证据：`src/lib/editor-data-storage.ts:591`、`:1206`、`:543`。

处理：

- `replaceFilesFromStaging` 后 fsync 每个目标目录。
- committed 状态恢复时读取 manifest/hash 并校验。
- 校验失败阻断启动恢复或进入 `needs-manual-repair` 状态。

### R2 明文备份策略

R2 自动备份对象继续是明文 JSON：

```json
{
  "schemaVersion": 1,
  "createdAt": "2026-06-08T00:00:00.000Z",
  "manifest": {},
  "articles": [],
  "navigation": [],
  "settings": {}
}
```

执行规则：

- 不增加远端口令字段。
- 不在 R2 object metadata 中放恢复所需秘密。
- latest key 固定，snapshot key 带时间、manifest hash、reason。
- 上传前计算 payload hash，上传后可选 head/get 验证。
- 下载后第一步必须 `JSON.parse`，再做 schema 校验。

### 本地备份策略

1. Docker 更新前：tar `.env data/`，证据：`deploy/git-deploy.sh:116`、`.github/workflows/docker-deploy.yml:374`。
2. 本地恢复前：新增持久 `pre-local-restore-<timestamp>.json`。
3. 远端恢复前：保持现有 `pre-remote-restore` snapshot。
4. 定时备份：每 3 小时 drain/同步，证据：`src/lib/startup-tasks.ts:27`、`:44`。
5. 手动导出：继续提供明文导出，独立工具链可以另有格式，但不得影响 R2 自动链路。

### manifest 校验

新增字段建议：

- `schemaVersion`
- `revision`
- `hash`
- `previousHash`
- `committedAt`
- `reason`
- `resources.articles.hash`
- `resources.navigation.hash`
- `resources.settings.hash`

借鉴：Pages CMS 的 `sha` 冲突检测、Tina 的 lock 校验、Decap 的 metadata 迁移，但保持本地 JSON 简单实现。

### 损坏文件处理

| 损坏对象 | 自动动作 | 用户可见状态 |
|---|---|---|
| `.backup-pending.json` | 移动到 `.backup-pending.corrupt-*`，重建空队列 | 高危：远端备份队列已重建，需要手动同步。 |
| restore state | 读取阶段判断 `staged/committed/failed` | 显示恢复卡住阶段和建议命令。 |
| `manifest.json` | 尝试从资源文件重建 hash，但不自动覆盖 | 进入需人工确认的修复流程。 |
| 单个资源 JSON | 保留坏文件，尝试从 latest 本地/R2 备份恢复 | 显示损坏资源名和最近可用备份。 |

### 灾备演练流程

每次实现数据安全改动后，至少演练：

1. 写入文章、导航、设置。
2. 触发 R2 latest 明文备份。
3. 下载 latest，确认可 `JSON.parse`。
4. 执行本地导出。
5. 模拟删除 `data/` 中一个资源。
6. 从本地导入恢复，确认 pre-local snapshot 已保留。
7. 从 R2 latest 恢复，确认 pre-remote snapshot 已保留。
8. 校验 manifest/hash/revision。
9. 运行相关测试：`tests/lib/r2-backup-storage.test.ts`、`tests/app/remote-backup-route.test.ts`、`tests/lib/backup-coordinator.test.ts`。

### 不适合照搬

- 不照搬 Pages CMS 的 Postgres 缓存和 GitHub App 权限。
- 不照搬 Tina 的 GraphQL bridge。
- 不照搬 Decap 多 backend 和 PR 工作流。
- 不引入数据库来解决个人站本地 JSON 的可恢复性问题。
- 不把 R2 自动备份改成不可直接阅读的格式。
