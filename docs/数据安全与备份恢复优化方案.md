## 一、当前数据安全链路与风险

### 1.1 数据存储链路（权威数据源 → 备份层级）

```
唯一权威数据源
  └─ 本地卷 /var/lib/blog-navigation/
      ├─ articles/articles.json          ← 文章
      ├─ navigation/tools.json           ← 导航
      ├─ settings/site.json              ← 设置
      ├─ settings/cloudflare-r2.json     ← R2 配置
      ├─ settings/editor-auth.json       ← 登录凭证
      ├─ manifest.json                   ← sha256 + revision 乐观锁
      └─ .restore-state.json             ← 恢复状态机（临时）

┌────────────── 默认风险层 ──────────────┐
│ 无异地备份（R2_BACKUP_ENABLED=false）│  ← P0-1
│ 单一本地卷，磁盘故障 = 全量丢失      │
└──────────────────────────────────────┘

备份通道 1：R2 远程备份（可选，默认关）
  └─ Cloudflare R2 Bucket
      ├─ latest/backup.json（明文）     ← P0-3
      └─ snapshots/YYYY/MM/DD/<ts>-<reason>.json（只增）  ← P1-5 无 GC

备份通道 2：GitHub 加密备份（手动，scripts）
  └─ scripts/data/runtime-data.mjs restore --encrypted
```

### 1.2 当前防护措施（值得肯定）

| 机制 | 位置 | 说明 |
|------|------|------|
| 原子写 | `editor-data-storage.ts:783-829` | temp + fsync + rename + dir fsync（Linux/macOS） |
| 文件锁 | `editor-data-lock.ts` | 进程内重入锁 + mkdir 目录锁（跨进程互斥但不跨容器） |
| manifest 乐观锁 | `editor-data-storage.ts:869-899` | sha256 + revision，写前比对冲突 409 |
| 崩溃恢复状态机 | `editor-data-storage.ts:1059-1139` | .restore-state.json 记录「replacing→committed」，启动自动恢复 |
| JSON 损坏保留现场 | `editor-data-storage.ts:1205-1246` | 抛 `EditorDataFileInvalidError` 不降级，人工介入 |
| navigation 区分损坏与缺失 | `markdown.ts:370-437` | 缺失回落种子，损坏抛错 |

> ✅ **这是在「不用数据库」前提下做得很好的持久化层**，对小规模单实例场景已达生产级。

### 1.3 **四个 P0 风险**（数据丢失维度）

#### P0-1：默认零异地备份
- **风险**：`R2_BACKUP_ENABLED=false` 是 README 部署默认值，单卷 `-v data:/var/lib/blog-navigation`，磁盘故障即全量丢失。
- **证据**：`README.md:254`、`README.md:117-125`、数据流 DL-8。
- **影响概率**：中（云盘年故障率 ~0.1-1%，单点无冗余）。
- **影响损失**：全部文章 + 导航 + 设置永久丢失。

#### P0-2：R2 远程备份 fire-and-forget 易丢
- **风险**：`queueCurrentBackupToRemote` 入队即返回，PUT 在后台。进程被 `docker rm -f` 或崩溃时，队列未 drain 则本次变更永不同步。
- **证据**：`src/lib/editor-remote-backup.ts:124-148`、`src/app/api/data/articles/route.ts:128`（DL-1）。
- **影响概率**：低（仅异常关闭时丢最后 0-N 次修改，但累积可能多条）。
- **影响损失**：部分内容丢失，用户可能感知不到漂移。

#### P0-3：R2 备份明文存储
- **风险**：`uploadBackupPayloadToR2` 直接 `JSON.stringify` 上传，文章/导航/settings（含 `editor-auth.json` 口令哈希、`cloudflare-r2.json` 凭证）全部明文落 R2。代码已埋 `securityWarning`。
- **证据**：`src/lib/r2-backup-storage.ts:142-156`、`486-538`（CR-3）。
- **影响概率**：低（需 R2 bucket 泄露或误配公开访问），但后果严重（凭证暴露）。
- **影响损失**：认证绕过 + 内容泄露。

#### P0-4：恢复事务回滚源未 fsync
- **风险**：`copyExistingFiles` 用 `fs.copyFileSync` 创建备份副本，无 fsync。崩溃时回滚目标可能截断，原子恢复的回滚保证失效（导致旧文件损坏但无法回滚）。
- **证据**：`src/lib/editor-data-storage.ts:1099-1110`、`515-526`（DL-5/RB-1）。
- **影响概率**：极低（需恰好在 copyFileSync 后、fsync 前崩溃）。
- **影响损失**：恢复中崩溃导致数据既不是新也不是旧，损坏且无回滚。

## 二、数据安全对比研究（外部项目实践）

### 2.1 备份层级（行业基线）

| 项目 | 本地备份 | 异地备份 | 自动定时 | 加密 | 版本保留策略 |
|------|---------|---------|----------|------|-------------|
| **node-fs-store** | ✅ 写入前多重 .bak（保留 3 份） | ❌ | ❌ | ❌ | FIFO 限 N |
| **electron-store** | ✅ 原子写 | ❌ | ❌ | ✅ 可选 AES | ❌ |
| **TinaCMS** | ✅ Git 即备份 | ✅ 推远程仓库 | ❌ | ❌ | Git 历史全保留 |
| **PouchDB** | ✅ MVCC 多版本 | ✅ 复制到 CouchDB | ⚠️ 连续复制 | ❌ | MVCC 压缩 |
| **blog-navigation** | ✅ restore 前备份 | ⚠️ R2（默认关、明文、fire-and-forget） | ❌ 无 cron | ⚠️ 脚本侧有但 R2 不用 | ❌ 快照只增 |

**结论**：
- ✅ **本地原子写与 manifest 完整性已达标**。
- ⚠️ **异地备份存在但默认关闭 + 不可靠 + 明文 + 无自动化**，低于生产基线。
- ⚠️ **快照只增不减，长期运行 R2 存储费持续增长**（1GB 约 $0.015/月，一年按天快照 ≈ 365 份）。

### 2.2 可借鉴点
1. **node-fs-store**：写入前多重备份（3 份带时间戳 .bak），FIFO 限数量。
2. **electron-store**：内置 AES-256-GCM 加密存储选项。
3. **proper-lockfile**：stale 阈值检测清理僵尸锁（本项目锁已有 heartbeat 类似机制）。
4. **TinaCMS**：Git 天然版本控制 + 分支工作流（不照搬，参考「差异对比能力」）。

## 三、优化方案（按 P0→P1 严重度）

### 3.1 P0-1：默认零异地备份 → 引导开启 R2 + 本地定时备份

#### 动作一：Setup 流程强化 R2 引导
- **改动**：`/setup` 页（`src/app/setup/page.tsx`）R2 配置区改为「必须配置或主动跳过」，跳过显示数据风险警告弹窗。
- **目标**：**将 R2 备份从「可选但默认关」改为「推荐且有引导」**，减少误操作零备份部署。
- **验证**：E2E Setup 测试包含配 R2 与跳过 R2 两路径；跳过路径确认有警告提示。

#### 动作二：本地定时备份脚本（Docker 内 cron 或外部）
- **位置**：`scripts/backup/local-snapshot.sh`（新增）。
- **行为**：定时（默认每 6 小时）`tar czf /backups/blog-data-$(date +%Y%m%d-%H%M%S).tar.gz /var/lib/blog-navigation`，保留最近 7 份 FIFO。
- **部署**：compose 增加 `/backups` 卷挂本地目录；可选 cron job 或 Docker HEALTHCHECK 前置脚本。
- **验证**：E2E 运行 24h 验证自动生成快照；手动损坏数据验证可从快照恢复。

#### 动作三：README 部署默认值改为引导配置
- **改动**：`README.md` Docker 运行示例加注释提示 R2 配置步骤，并链接到「备份与恢复」专章（新增 `docs/BACKUP_AND_RECOVERY.md`）。
- **目标**：降低「以为只是本地跑跑」导致生产零备份。

---

### 3.2 P0-2：R2 备份持久化队列（防 fire-and-forget 丢）

#### 问题根源
`queueCurrentBackupToRemote` 入队后返回，队列在进程内存。异常退出时队列丢失，上传永不发生。

#### 方案：落盘 pending + drain
```
编辑器 API 写成功
  → BackupCoordinator.enqueue
      ① 落盘 pending.json（原子写）          ← 新增
      ② 入内存队列（单飞合并）              ← 现有
  → 返回
后台 drain 循环（process 启动时启动）
  ① 扫 pending.json 有未完成任务
  ② 上传 R2（明文 JSON，见 §3.3）
  ③ 成功后删 pending 条目（原子更新）
  ④ 失败记日志 + 指数退避重试（限 3 次）
```

- **涉及**：`src/lib/editor-remote-backup.ts:55-148`、新增 `src/lib/backup-coordinator.ts`。
- **兼容**：进程内队列保留，pending.json 仅作持久化保底；优雅关闭 `SIGTERM` 时 drain 队列再退出（timeout 30s）。
- **验证**：注入「drain 前 kill -9」测试，重启后自动上传 pending；注入「R2 PUT 失败」测试重试逻辑。
- **工作量**：1.5-2 天。

---

### 3.3 P0-3：R2 明文 JSON 备份约定

#### 动作
R2 远端备份明确使用明文 JSON，避免恢复时依赖额外口令：
- **配置**：R2 配置只包含连接凭证和对象前缀，不增加恢复所需的额外密钥字段。
- **关键改动**：`uploadBackupPayloadToR2` 直接上传格式化 JSON；`downloadLatestBackupPayloadFromR2` 直接解析 JSON。
- **验证**：E2E 上传明文备份 → 下载恢复 → 数据完整。
- **工作量**：0.5-1 天。

---

### 3.4 P0-4：恢复前备份加 fsync

#### 动作
`copyExistingFiles`（`editor-data-storage.ts:1099-1110`）拷贝后对每个备份文件显式 fsync；Windows 兼容（已有 `fsyncDirectory` 处理）。
```ts
for (const file of filesToBackup) {
  fs.copyFileSync(file, backupPath)
  const fd = fs.openSync(backupPath, 'r+')
  fs.fsyncSync(fd)                         // ← 新增
  fs.closeSync(fd)
}
```
- **涉及**：`editor-data-storage.ts:1099-1110`。
- **验证**：单元测试模拟崩溃时序（难），代码审查 + 类比现有 fsync 测试覆盖。
- **工作量**：0.5 天。

### 3.5 P1-5：R2 快照 GFS 保留策略

#### 问题
`createSnapshotBackupKey` 按年/月/日写 `snapshots/YYYY/MM/DD/<ts>-<reason>.json`，全代码无 ListObjects/DeleteObject。

#### 方案：GFS（Grandfather-Father-Son）
- **保留规则**：每日快照保留最近 7 天；每周日快照保留最近 4 周；每月 1 号快照保留最近 12 月。
- **清理时机**：每次上传快照后触发 GC（异步，不阻塞）。
- **实现**：新增 `cleanupOldSnapshots`（`r2-backup-storage.ts`），用 `listObjectsV2` 遍历 `snapshots/` 前缀，按 key 解析日期，删除不在保留窗口的对象。
- **配置**：`R2_SNAPSHOT_RETENTION_DAYS`/`WEEKS`/`MONTHS`（默认 7/4/12）。
- **验证**：生成 100 个快照 fixture，运行 GC，确认保留 7+4+12=23 份左右（周日/月初重叠可能少于 23）。
- **工作量**：1-1.5 天。

### 3.6 P1-9：自动 / 定时备份

#### 方案一：GitHub Actions schedule（推荐）
新增 `.github/workflows/scheduled-backup.yml`：
```yaml
on:
  schedule:
    - cron: '0 */6 * * *'  # 每 6 小时
  workflow_dispatch:        # 手动触发
jobs:
  backup:
    runs-on: ubuntu-latest
    steps:
      - name: Trigger remote backup
        run: |
          curl -X POST https://${{ secrets.PROD_DOMAIN }}/api/data/backup \
            -H "Authorization: Bearer ${{ secrets.EDITOR_TOKEN }}" \
            -H "X-CSRF-Token: ${{ secrets.CSRF_TOKEN }}"
```
- **优点**：无需 Docker 内 cron，利用 GitHub 定时触发；失败有通知。
- **缺点**：依赖 GitHub Actions 配额；需配 secrets。

#### 方案二：容器内 supercronic（备选）
- Dockerfile 加 `supercronic`（无 root cron daemon 替代）。
- `scripts/backup/crontab` 定义 `0 */6 * * * /app/scripts/backup/trigger-remote.sh`。
- **优点**：自包含，不依赖外部服务。
- **缺点**：增加镜像复杂度。

**推荐**：方案一（GitHub Actions），保持镜像简单。

## 四、灾备演练流程（可验证性）

> 参考 docker_cicd agent 结论：演练是检验备份可靠性的唯一手段，必须定期执行（建议季度 1 次）。

### 4.1 演练清单
1. **场景一：本地卷损坏**
   - ① 停容器；② 删 data 卷；③ 从 R2 latest 恢复；④ 验证文章/导航/设置完整；⑤ 时序：10 分钟内完成。
2. **场景二：R2 latest 与 snapshot 不一致**
   - ① 手动损坏 latest；② 从最近 snapshot 恢复；③ 验证完整性。
3. **场景三：完整重建**
   - ① 新机器部署；② 仅挂空卷；③ 填 R2 配置；④ API 触发恢复；⑤ 验证。
4. **场景四：GitHub 加密备份恢复**
   - ① `scripts/data/runtime-data.mjs restore --encrypted --backup-file=xxx`；② 验证。

### 4.2 演练自动化
- **脚本**：`scripts/test/disaster-recovery-drill.sh`（新增），自动化上述 4 场景，输出通过/失败报告。
- **集成**：可选加入 CI（但演练需真实 R2 凭证，建议手动 `workflow_dispatch` 触发）。
- **文档**：`docs/DISASTER_RECOVERY_PLAYBOOK.md`（新增），记录每次演练时间、结果、发现问题。

## 五、数据安全检查清单（运行时 & 部署时）

### 运行时检查（启动时自检）
- [ ] R2 备份开启且连接配置完整（否则 WARN 级日志）
- [ ] pending.json 存在未完成任务 → 自动 drain
- [ ] manifest.json 与数据文件 hash 一致（不一致自动重建 manifest + ERROR 日志）
- [ ] editor-auth.json 存在且哈希算法为 scrypt（弱哈希告警）
- [ ] 磁盘剩余空间 > 1GB（低于阈值拒绝写入 + 503）

### 部署时检查（CI/smoke）
- [ ] Docker 镜像 Trivy 扫描无 HIGH/CRITICAL CVE
- [ ] compose 配置包含 read_only + tmpfs + cap_drop
- [ ] HEALTHCHECK 配置存在
- [ ] R2 备份配置已填写（生产）或有跳过确认文档

## 六、成功标准
- 默认部署路径**引导用户配置 R2 或明确跳过**，不再静默零备份；
- R2 备份队列持久化，异常退出后自动 drain 补传；
- R2 备份保持**明文 JSON**，下载后可直接解析恢复；
- 恢复前备份文件 fsync 到位，回滚源可靠；
- 快照按 GFS 自动清理，存储费可控；
- 灾备演练脚本就绪，文档记录每次演练结果；
- 季度灾备演练通过率 100%。
